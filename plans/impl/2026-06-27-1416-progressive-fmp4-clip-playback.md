# Plan: Progressive fMP4 clip playback while pulling

## Context

Today the clip viewer plays a recorded clip in three strictly-sequential steps
(ADR `app/docs/design/07-2026-06-26-on-device-clip-remux-playback.md`):

1. `ClipPullClient` pulls the **entire** `.ts` from the Pi to a temp file (resumable
   `Range`/`If-Range`, `raspi/service/src/clips.rs#serve_clip`).
2. `ClipRemuxer.live` remuxes the finished `.ts` to a normal `.mp4` with `AVAssetWriter`
   (whole-file: `TSDemuxer.demuxH264(from:)` does `Data(contentsOf:)`).
3. `ClipViewerViewController` plays the MP4 with `AVPlayer(url:)`.

The clip is only watchable after **both** the pull and the remux finish. On the real-Pi
baseline (2026-06-26, iPhone `Pelucho`, `seg_00084.ts` over `dancam-dev`:
`38.5 MB pulled - 37.5 MB playable - 9.1 s - 34 Mbps`) the user stares at a progress bar
for ~9 s before the first frame. A 30 s segment is ~38 MB at 10 Mbps CBR, so this only
gets worse on a congested link.

**Goal:** start playback **before** the full `.ts` download completes, and once the pull
finishes give the user a properly scrubbable clip -- without moving any work to the Pi,
without AVPlayer ever touching the Pi, and without weakening crash-safe TS recording.

**Approach (decided):** generate **fragmented MP4 (fMP4/CMAF)** from the `.ts` bytes as
they land on disk, and feed AVPlayer a **local HLS playlist served by a loopback HTTP
server bound to 127.0.0.1** (EVENT while pulling). When the pull completes, run the
existing whole-file remux to produce a durable, front-`moov` normal `.mp4`, and **swap**
the player to that file for precise scrubbing / cache / future export. The whole-file
remux path is retained unchanged as both the finalizer **and** the fallback, so the
progressive path is strictly additive: if anything progressive fails, the viewer behaves
exactly as it does today.

This is the "riskiest" / "chunky" app step of roadmap swoop `lime`
(`docs/roadmap.md#Swoop lime -- Watch recorded clips`).

## Why these choices (load-bearing decisions)

### Not classic faststart MP4 for early play
Faststart (front `moov`) only helps once the **complete** sample tables are known. While
remuxing a `.ts` as it downloads, those tables are not known, so faststart cannot be the
early-play strategy. Faststart **is** correct for the finalized durable MP4 (set
`shouldOptimizeForNetworkUse = true`), because by then the whole file is known. No
contradiction: early play uses fMP4 fragments; the final artifact uses faststart MP4.

### fMP4 is generated by AVAssetWriter in HLS segmenting mode (verified), not hand-rolled
iOS has a first-class API for emitting fMP4 segments, available since iOS 14
(WWDC20 "Author fragmented MPEG-4 content with AVAssetWriter"):
`AVAssetWriter(contentType: .mpeg4Movie)` + `outputFileTypeProfile = .mpeg4AppleHLS` +
`preferredOutputSegmentInterval = .indefinite` + `flushSegment()` +
`AVAssetWriterDelegate.assetWriter(_:didOutputSegmentData:segmentType:segmentReport:)`.
The delegate hands back the **initialization segment once** (`.initialization` =
`ftyp`+`moov` with `mvex`/`trex`) then **one media segment per flush** (`.separable` =
`styp`+`moof`+`mdat`). This reuses the exact CoreMedia code already proven in
`ClipRemuxerEngine` (`makeFormatDescription`, `makeSampleBuffer`); we add only the writer
configuration and the delegate. A hand-rolled box writer is a documented escape hatch,
not a planned chunk.

### Delivery is a loopback HLS server, not a resource loader (verified + user-confirmed)
- AVPlayer **rejects** raw HLS media-segment bytes returned from
  `AVAssetResourceLoaderDelegate` (`CoreMediaErrorDomain -12881 "custom url not redirect"`);
  segments must be fetched over real HTTP. So a no-server HLS path is a non-starter.
- The single-growing-fMP4-as-one-resource loader path is documented-fragile for
  unknown/growing length (the player can stop issuing data requests mid-stream). We do
  not know the final fMP4 size up front, which puts us squarely in that risky regime.
- HLS **EVENT** playlists are purpose-built for "content produced live, played from the
  start, grows over time": append segments as they flush, write `#EXT-X-ENDLIST` only at
  the end. AVPlayer's HLS engine handles playlist reload, buffering, and in-buffer seek
  natively.
- The loopback server was deleted in ADR 07 for a **representation** reason
  (single-segment TS HLS scrubbed badly), not a mechanism reason. Serving multi-segment
  fMP4 with real per-segment durations, plus swapping to the finalized MP4 for scrubbing,
  removes that objection. 127.0.0.1 never leaves the device, so "AVPlayer never talks to
  the Pi" holds.

**User-mandated hard requirements for the server** (carry verbatim into ADR 08):
- Bind **only** to loopback (127.0.0.1 / `localhost`); never `0.0.0.0`.
- Server lifetime **scoped to the viewer/session**; torn down on swap and on dismiss.
- Playlist `#EXTINF` durations come from **`AVAssetSegmentReport`**, not guessed.
- `flushSegment()` **only at IDR boundaries** (one segment per GOP, each starting on a
  sync sample).
- Once the full clip is available, **switch to / cache a normal finalized MP4** for
  scrubbing and export.
- **Write a new ADR** (08) superseding/amending ADR 07; do not quietly reintroduce the
  server. The ADR note must be narrow: the prior problem was single-segment TS HLS, not
  localhost HLS itself.

### File-as-buffer seam, not a `.body(Data)` event
`ClipPullClient`'s stream is an unbounded `AsyncThrowingStream`; `continuation.yield`
never suspends the producer. A `.body(Data)` event therefore cannot exert backpressure --
a segmenter slower than the network would queue the whole ~38 MB clip in memory (on top
of the disk copy), and the only built-in bound (`.bufferingNewest`) **drops** bytes,
which is fatal for a byte-exact `.ts`. Instead: the bytes already land on disk in order
via `FileHandle.write` in `ClipPullClient.swift#writeDecodedChunks`, and
`.progress(bytesWritten:)` is already a precise, monotonic "bytes < N are flushed and
durable" signal. So we add **one** event, `.opened(fileURL:)`, and reuse `.progress` as
the availability signal; the segmenter reads `[lastRead, bytesWritten)` from the `.ts`
with its own read handle. Lossless, zero duplication, no backpressure. Truncation
(200-on-resume) carries an **explicit `.restarted` event** (see section 6), not an inferred
`bytesWritten` rewind: `ClipPullClient` zeroes `bytesWritten` *silently* in
`prepareBodyDecoder`'s `case 200:` and only yields the next `.progress` after replacement
bytes land, so an early truncation can make the first post-reset `.progress` >= the
pre-truncation watermark -- a rewind-based detector would miss it and splice new TS bytes
onto stale read-ahead. `.restarted` is **terminal** for the progressive attempt (it cannot
safely continue an append-only EVENT timeline across a representation change); the pull
still completes and the finalizer serves durable playback (sections 6, 7, 9).

### Accept a double parse for now
The streaming pipeline parses the `.ts` for fMP4 segments; the finalizer parses the whole
`.ts` again for the durable MP4. The cost is TS framing + Annex-B split done twice over a
short clip (CPU-bound, sub-second to low-seconds; bytes already on disk). The benefit is
that the durable artifact rides the **already-proven, fully-tested** whole-file path,
fully decoupled from the newer streaming code, so a subtle streaming bug cannot corrupt
the cache/export artifact. A future optimization ("single parse -> two writers") is noted
once the streaming path is battle-tested. Out of scope here.

## Architecture overview

```
ClipPullClient (.live)                ProgressiveSegmenter (.live)            ClipViewerViewController (@MainActor)
  pull .ts over Range/If-Range          (off-main; owns the progressive rep)     state machine + AVPlayer + swap
  writeDecodedChunks -> FileHandle      reads [lastRead, avail) from .ts
  yields:                               IncrementalTSDemuxer  -> PES
    .opened(fileURL) ----------------->   StreamingH264AccessUnitAssembler -> AUs
    .progress(bytesWritten) --(avail)->   FMP4Segmenter (AVAssetWriter HLS seg) -> init + per-GOP segments
    .completed(ClipPullResult)            LoopbackMediaServer (127.0.0.1)  -> p.m3u8 / init.mp4 / segN.m4s
                                          emits: .firstPlayableReady(playlistURL) -----> AVPlayer(url: playlistURL), play()
  on .completed:                                                                         (playingWhilePulling)
    ClipRemuxer.live.remux(.ts) -> durable front-moov .mp4 ----------------------------> replaceCurrentItem + seek(currentTime)
                                                                                         (readyScrubbable; tear down server)
```

The finalizer (`ClipRemuxer.live`) **always** runs on `.completed` regardless of whether
progressive playback ever started, which is what makes the fallback inherent.

## Detailed design

### 1. Incremental TS demuxer

New `IncrementalTSDemuxer` alongside the existing `TSDemuxer` (which stays, unchanged in
behavior, for the finalizer/fallback). Extract the per-packet / PSI / PES parsing that
currently lives inside `TSDemuxer.demuxH264(from:Data)` into shared `static` helpers so
both the whole-file and incremental parsers share one implementation of record. Reuse
`parsePAT`, `parsePMT`, `psiSection`, `parsePESStart`, `decodeTimestamp`, and the private
`PartialPES`.

```swift
nonisolated struct IncrementalTSDemuxer {
    init(clockTimescale: Int32 = 90_000)

    /// Append an arbitrary byte chunk (any size, not 188-aligned). Buffers a
    /// partial-packet residual; persists PAT/PMT/PID + partial-PES across calls.
    /// Returns the PES packets that *finalized* during this append (a video PES is
    /// finalized when the next payload-unit-start arrives on the video PID).
    mutating func append(_ chunk: Data) throws -> [H264PESPacket]

    /// Flush the last in-flight PES. Call once when the pull completes.
    mutating func finish() -> [H264PESPacket]
}
```

State carried across `append`: `residual: Data` (trailing bytes of an incomplete 188-byte
packet), `pmtPID`/`videoPID` (latched once seen), `currentPES: PartialPES?`, `synced`.
Logic per `append`: prepend `residual`, process complete 188-byte packets (require sync
byte `0x47`; the Pi file is packet-aligned from byte 0), route PAT -> PMT -> videoPID
exactly as today, finalize `currentPES` on `payloadUnitStart`, keep the tail as the new
`residual`. Add **resync robustness**: if the sync byte is missing, scan forward for an
offset where bytes at `+0/+188/+376` are all `0x47` (triple-lock) before resuming; emit a
one-time log if resync fires. `TSDemuxer.demuxH264(from:Data)` keeps its strict
`% 188 == 0` guard (fail-loud for the finalizer) but calls the same shared helpers.

Files: `app/DanCam/DanCam/Media/Remux/TSDemuxer.swift` (extract helpers + add type).

### 2. Streaming access-unit assembly

New `StreamingH264AccessUnitAssembler` that computes durations with **one-access-unit
lookahead** (no whole-file sort / no whole-file median). Promote
`splitAnnexB`, `splitAccessUnitGroups`, `avccSampleData`, `isSliceNAL`, `findStartCode`
in `H264AccessUnitAssembler.swift` to shared `static` helpers used by both the batch
assembler (unchanged behavior) and the streaming one.

```swift
nonisolated struct StreamingH264AccessUnitAssembler {
    init(timescale: Int32 = 90_000, frameDurationWindow: Int = 32)

    struct Output: Sendable {
        var accessUnits: [H264AccessUnit]   // newly finalized, strict decode order
        var sps: Data?                      // non-nil only on the append that first sees it
        var pps: Data?
        var didBecomeReady: Bool            // true on the append where SPS+PPS both first known
    }

    /// Feed PES finalized by the demuxer (assumed DTS/decode order for a single
    /// H.264 stream). Latches SPS/PPS; emits AU[i] once AU[i+1]'s DTS is known
    /// (duration = next.dts - this.dts).
    mutating func append(_ packets: [H264PESPacket]) throws -> Output

    /// Emit the final held AU using the running frame-duration estimate.
    mutating func finish() throws -> Output
}
```

State: latched `sps`/`pps`; a single `held` pending AU whose duration is unresolved until
the next AU arrives; a bounded window of recent DTS deltas for a **rolling median**
(matches today's whole-file median; fallback `3_000` ticks like
`H264AccessUnitAssembler#inferredFrameDuration`). Per PES: split NALs, latch SPS(7)/PPS(8),
group by AUD, build slice-bearing pending AU(s) (`avccSampleData`, `isKeyFrame` =
contains type 5). Pipeline through `held`: when AU `n` arrives, finalize `held` with
`duration = n.dts - held.dts`, push the delta into the window, emit, set `held = n`.
**Guard `duration > 0`** -> else throw `ClipRemuxError.invalidH264("DTS not strictly
increasing")`; this strict check *is* the validation of the "PES arrive in decode order"
assumption (the Pi stream has no B-frames, so DTS is monotonic). Do **not** emit any AU
before SPS+PPS are latched (the segmenter needs the format description first). Rare
multi-AU-per-PES: defer subdivision until the next PES DTS is known (streaming analogue of
`inferredPacketDuration`); guard/log because the recording facts say one frame per PES.
`finish()` finalizes `held` with the rolling-median duration.

Files: `app/DanCam/DanCam/Media/Remux/H264AccessUnitAssembler.swift`.

### 3. fMP4 segmenter (AVAssetWriter HLS segmenting)

```swift
// Sink methods are called in publication order (init, then seg0, seg1, ...). The
// implementer confines all state to one serial domain (section 4) and assigns segment
// indices there, so segment callbacks arriving on the writer's thread cannot race the
// HTTP reader.
protocol FMP4SegmentSink: AnyObject, Sendable {              // the loopback server implements this
    func appendInitializationSegment(_ data: Data)
    func appendMediaSegment(_ data: Data, duration: CMTime)
    func finish()                                            // pull complete -> close playlist (ENDLIST)
}

nonisolated final class FMP4Segmenter: NSObject, AVAssetWriterDelegate {
    init(timescale: Int32 = 90_000, sink: FMP4SegmentSink)
    func start(sps: Data, pps: Data) throws                  // builds the writer lazily on first AU
    func append(_ accessUnit: H264AccessUnit) throws         // flushSegment() before each non-first IDR
    func finishWriting() throws                              // flush trailing GOP; markAsFinished; semaphore-backed finish; sink.finish()
}
```

Writer setup (mirrors `ClipRemuxerEngine` but URL-less + segmenting), reusing
`makeFormatDescription` and `makeSampleBuffer`:

```swift
let writer = AVAssetWriter(contentType: .mpeg4Movie)
writer.outputFileTypeProfile = .mpeg4AppleHLS
writer.preferredOutputSegmentInterval = .indefinite     // manual control; flush at IDR
writer.initialSegmentStartTime = .zero
writer.delegate = self
let input = AVAssetWriterInput(mediaType: .video, outputSettings: nil,
                               sourceFormatHint: formatDescription)  // nalUnitHeaderLength: 4
input.expectsMediaDataInRealTime = false
writer.add(input); writer.startWriting(); writer.startSession(atSourceTime: .zero)
```

Per-AU control flow in `append`: if `accessUnit.isKeyFrame` and >= 1 sample already
appended, call `writer.flushSegment()` **before** appending the IDR sample (closes the
previous GOP into a `.separable` segment; the flush-before-sync-sample order avoids the
"fragmented media data started with nonsync sample" error). Build the `CMSampleBuffer`
with the existing `makeSampleBuffer` (timing + `kCMSampleAttachmentKey_NotSync`), spin-wait
on `isReadyForMoreMediaData` (the established pattern). **Rebase PTS/DTS to zero** by
subtracting `firstDecodeTicks` before building sample buffers (see Finalize + swap for why).
Both `append` and `finishWriting` are **synchronous** (`throws`, no `async`): `append` already
blocks on the spin-wait, and `finishWriting` wraps `AVAssetWriter.finishWriting`'s completion
handler in a `DispatchSemaphore` exactly as `ClipRemuxerEngine.finish` does -- so the whole
segmenter is driven inline on section 5's dedicated serial queue with no `await` and no
cooperative-pool hop.

Delegate. `AVAssetWriterDelegate.assetWriter(_:didOutputSegmentData:...)` may be invoked on
a thread other than the one calling `flushSegment()`, so the callback **copies** the
segment bytes and hands them to the sink, which applies appends in publication order on its
single serial domain (section 4). No EXTINF guessing: a `.separable` segment whose
`AVAssetSegmentReport` is missing or whose track duration is non-numeric/non-positive
**fails the progressive path** (it never publishes a guessed duration) and routes to the
finalized-MP4 fallback (section 9) -- honoring the user requirement that durations come from
`AVAssetSegmentReport`.

```swift
func assetWriter(_ w: AVAssetWriter, didOutputSegmentData data: Data,
                 segmentType: AVAssetSegmentType, segmentReport: AVAssetSegmentReport?) {
    let bytes = Data(data)   // copy: this callback may run off the segmenter's thread
    switch segmentType {
    case .initialization:
        sink.appendInitializationSegment(bytes)              // sink enqueues on its serial domain
    case .separable:
        guard let dur = segmentReport?.trackReports.first?.duration,
              dur.isNumeric, dur.seconds > 0 else {
            failProgressive(.missingOrInvalidSegmentReport); return   // -> fallback, never a guess
        }
        sink.appendMediaSegment(bytes, duration: dur)        // sink assigns the index, in order
    @unknown default: break
    }
}
```

The delegate cannot throw, so `failProgressive(_:)` latches a terminal error that the next
`append(_:)` / `finishWriting()` re-throws, surfacing on the `ProgressiveSegmenter` event
stream -> `handleProgressiveFailure`. The delegate may run on a different thread than
`append(_:)`/`finishWriting()`, so the latch is **lock-guarded** (a `Mutex` from
`Synchronization`, or `OSAllocatedUnfairLock`), never a bare `var` -- the write (delegate
thread) and the read (pipeline context) are a genuine cross-thread hand-off that a
strict-concurrency build would otherwise flag. Files: new
`app/DanCam/DanCam/Media/Stream/FMP4Segmenter.swift`; touch `ClipRemuxerEngine.swift`
(extract `makeFormatDescription`/`makeSampleBuffer`/`cmTime` for reuse).

### 4. Loopback HLS server (delivery)

```swift
nonisolated final class LoopbackMediaServer: FMP4SegmentSink {
    init() throws                      // NWListener bound to 127.0.0.1 on an OS-assigned port
    var mediaPlaylistURL: URL { get }  // http://localhost:<port>/p.m3u8
    // FMP4SegmentSink (every method mutates state on one serial queue the HTTP handler also reads on):
    func appendInitializationSegment(_ data: Data)               // -> /init.mp4
    func appendMediaSegment(_ data: Data, duration: CMTime)      // -> /segN.m4s (index assigned here), append EXTINF
    func finish()                                                // append #EXT-X-ENDLIST
    func shutdown()                                              // close listener; drop temp segment dir
}
```

Three routes, served from a path map whose segment bodies spill to a temp dir to bound
memory on long clips. **All server state -- the playlist string, the path map, and the
running segment index -- is confined to one serial `DispatchQueue`.** Sink appends run on
that queue (the section-3 delegate already copied the bytes); the `NWListener` connection
handler also parses/serves requests on that same queue, so a request can never observe a
half-updated playlist and segment indices never race. `.firstPlayableReady` and `finish()`
are signaled from that queue **after** the triggering append has applied (init + seg0
present; or ENDLIST written).

- `GET /p.m3u8` -> `application/vnd.apple.mpegurl`, a growing **EVENT** media playlist:
  ```
  #EXTM3U
  #EXT-X-VERSION:7
  #EXT-X-TARGETDURATION:<frozen at first segment; never rewritten>
  #EXT-X-MEDIA-SEQUENCE:0
  #EXT-X-PLAYLIST-TYPE:EVENT
  #EXT-X-MAP:URI="init.mp4"
  #EXTINF:<exact dur from AVAssetSegmentReport>,
  seg0.m4s
  ...            (append per flushed segment; add #EXT-X-ENDLIST on finish())
  ```
  **`#EXT-X-TARGETDURATION` is frozen and never changes** (RFC 8216 6.2.1: "The value of
  the EXT-X-TARGETDURATION tag ... MUST NOT change"; and every `#EXTINF`, rounded, MUST be
  <= it). It is frozen when the first media segment is appended -- which is before
  `mediaPlaylistURL` is ever handed to AVPlayer (at `.firstPlayableReady`) -- to a
  conservative GOP-derived ceiling `max(configured ceiling, ceil(first #EXTINF) + margin)`.
  The Pi runs a fixed keyframe interval, so GOP durations are near-constant and the
  trailing GOP is shorter, not longer; the ceiling is kept tight to keep AVPlayer's EVENT
  reload cadence brisk. If any later segment's reported `#EXTINF` would round above the
  frozen ceiling, that is a progressive failure -> fall back to the finalized MP4 (section
  9); the header is never silently mutated. Per-segment `#EXTINF` values remain the exact
  `AVAssetSegmentReport` durations (section 3) -- the frozen ceiling is only the header
  bound, never a source of guessed per-segment timing.
- `GET /init.mp4` -> `video/mp4` (the `.initialization` data).
- `GET /segN.m4s` -> `video/iso.segment` (each `.separable` data).

No master playlist (single video-only rendition); AVPlayer loads `mediaPlaylistURL`
directly. The responder handles **`GET`, `HEAD`, and a single `Range: bytes=a-b`** on every
route: AVPlayer probes local HTTP media with HEAD and byte-range requests, and the prior
loopback server already supported both, so dropping them risks early play breaking on a
transport detail unrelated to fMP4. `HEAD` returns status + headers (`Content-Length`,
`Content-Type`, `Accept-Ranges: bytes`) with no body; a satisfiable range returns `206`
with `Content-Range`; an unsatisfiable range returns `416`. An earlier `LoopbackHLSServer`
(deleted in `b384759`) can be recovered from git history as NWListener + HTTP/1.1
scaffolding (it already did HEAD + ranges), but the new one serves fMP4 + a live/VOD
playlist rather than a single TS. **ATS:** loopback is exempt from the Local Network
privacy prompt; for cleartext to localhost, prefer hostname `localhost` and add
`NSAppTransportSecurity -> NSAllowsLocalNetworking = true` (scoped). Files: new
`app/DanCam/DanCam/Media/Stream/LoopbackMediaServer.swift`; possibly `Info.plist`.

### 5. ProgressiveSegmenter dependency seam

One struct-of-closures dependency mirroring `ClipRemuxer`, registered in `AppDependencies`
(default `.noop`, source-compatible). It owns the demuxer + assembler + segmenter + server
off the MainActor and **emits a local URL, not an `AVPlayerItem`** (the latter is not
`Sendable` and wants the MainActor). The viewer owns the `AVPlayer` and the swap.

```swift
nonisolated enum ProgressiveSegmenterEvent: Equatable, Sendable {
    case opened(workDirectory: URL)         // temp dir the viewer must delete on teardown
    case firstPlayableReady(url: URL)       // init + first GOP exist -> AVPlayer can open `url`
}

nonisolated struct ProgressiveSegmenter: Sendable {
    // `availability` carries monotonically-increasing bytes-available [0, N). A pull
    // truncation never appears here: the viewer treats it as terminal, cancels this
    // stream, and falls back (section 7/9), so the segmenter has no reset/re-read path.
    var start: @Sendable (
        _ sourceURL: URL, _ clipID: Int,
        _ availability: AsyncStream<UInt64>
    ) -> AsyncThrowingStream<ProgressiveSegmenterEvent, Error>
    static let live: ProgressiveSegmenter
    static let noop: ProgressiveSegmenter   // drains availability; never produces a playable
}
```

`.live` drives the blocking demux -> assemble -> segment pipeline on a **dedicated serial
`DispatchQueue` -- genuinely off the Swift cooperative pool** (GCD gives the queue its own
thread), **not an `actor` and not a detached `Task`**: the segmenter's
`isReadyForMoreMediaData` + `Thread.sleep(0.002)` spin-wait (section 3) blocks whatever thread
it runs on, and both an `actor` and a `Task.detached` would block a *cooperative-pool* thread
(a detached task is unstructured but still runs on the cooperative pool -- it is not off-pool),
risking starvation / priority inversion exactly in the spike-2 "pull concurrent with live
preview" case this isolation exists to protect. The finalizer (`ClipRemuxerEngine.remux`) gets
away with `Task.detached` only because its spin-wait is a one-shot whole-file write that rarely
engages; the segmenter is fed incrementally under concurrent load, so it takes the stronger
isolation of a real serial queue. On each availability value `N`, the pipeline reads `[lastRead, N)` from
`sourceURL` with its own read `FileHandle`, feeds `IncrementalTSDemuxer` ->
`StreamingH264AccessUnitAssembler` -> `FMP4Segmenter` (-> `LoopbackMediaServer`), and emits
`.firstPlayableReady(mediaPlaylistURL)` once the init segment + first media segment exist.
The `availability` `AsyncStream` and the `AsyncThrowingStream` continuation are the only
cross-context hand-offs (both `Sendable`). This pipeline context is **distinct from** the
loopback server's own serial queue (section 4): the blocking append never runs on the queue
that serves AVPlayer's HTTP requests, so segment production cannot stall playlist / segment
delivery. There is **no** mid-stream reset/re-read path: a pull truncation is terminal for
the progressive attempt (section 7/9), so the viewer cancels this stream and falls back to
the finalizer rather than asking the segmenter to rebuild.

No `extended` / buffered-duration event and no `finished` / EOF event: scrub-before-completion
is served natively by AVPlayer reloading the growing EVENT playlist, and `playingWhilePulling`
already draws its progress from the pull's `.progress` bytes -- an `extended` event would be
seam surface with no consumer. The segmenter's end-of-work likewise needs no event: the swap is
driven by the pull's `.completed`, ENDLIST is written internally via
`FMP4Segmenter.finishWriting` -> `sink.finish()` (AVPlayer learns the EVENT timeline is complete
from ENDLIST natively), and the `AsyncThrowingStream` simply finishing already signals clean EOF
(a normal loop exit, distinct from a thrown failure). Both are omitted until a viewer feature
needs an explicit buffered-range or end-of-segmentation readout.
`AppDependencies`: add `var progressiveSegmenter: ProgressiveSegmenter`
(memberwise default `.noop`; `configuration:` init sets `.live`). Files: new
`app/DanCam/DanCam/Media/Stream/ProgressiveSegmenter.swift`;
`app/DanCam/DanCam/App/AppDependencies.swift`.

### 6. ClipPullClient `.opened` seam

```swift
nonisolated enum ClipPullEvent: Equatable, Sendable {
    case opened(fileURL: URL)                                // NEW -- fires exactly once
    case restarted                                          // NEW -- file truncated to 0; discard read-ahead
    case progress(bytesWritten: UInt64, expected: UInt64?)  // unchanged (doubles as availability)
    case completed(ClipPullResult)                          // unchanged
}
```

`.opened` fires once in `ClipPullClient.swift#producePull`, immediately after the write
`FileHandle` is opened and **above** the `attempts:` retry loop, carrying the same
`outputURL` that later appears in `.completed`:

```swift
let outputHandle = try FileHandle(forWritingTo: outputURL)
fileHandle = outputHandle
continuation.yield(.opened(fileURL: outputURL))   // INSERT HERE -- exactly once
```

`.opened` guarantees: exactly once (above the loop, so retries/206/200-truncate/416 never
re-emit); ordered before every `.progress` and `.completed`; the file exists (empty) when
it fires.

`.restarted` is yielded from `prepareBodyDecoder`'s `case 200:` (validator-change
truncation) **after** `fileHandle.truncate(atOffset: 0)` / `bytesWritten = 0` and
**before** the caller's `writeDecodedChunks` writes any replacement bytes -- i.e. ahead of
the next `.progress`. (`prepareBodyDecoder` does not currently take the continuation;
thread it in, or have it return a `didTruncate` flag and let `runAttempt` yield
`.restarted` between `prepareBodyDecoder` and `writeDecodedChunks`.) This and `.opened` are
the **only** behavioral changes to the pull body; `runAttempt`/`makeRequest`/
`writeDecodedChunks` and the 200/206/416, offset/short-end, and validator-change branches
are otherwise unchanged, and `.completed` still fires with the `.ts` fully written.

Why an explicit event and not a `.progress` rewind: on a `case 200:` truncation
`bytesWritten` is reset to 0 *silently* and the next `.progress` only fires once
replacement bytes land, so if the truncation happened early (small pre-truncation
watermark) the first post-reset `.progress` can be **>=** that watermark -- a rewind-based
detector would miss it and the segmenter would splice new TS bytes onto stale read-ahead.
`.restarted` makes the change unambiguous, and the viewer treats it as **terminal** for the
progressive attempt (section 7/9): a validator change means the on-disk bytes are now a
different representation, and an EVENT playlist's already-served segments cannot be
rewritten or mixed, so there is no safe way to continue the same timeline. The pull itself
still re-pulls from 0 and reaches `.completed`; the always-on finalizer then builds the
durable MP4 from the corrected `.ts`, so durable playback is unaffected.

### 7. Viewer state machine + orchestration

Replace the implicit flow in `ClipViewerViewController.runPull` with an explicit
`enum ViewerState` and a single `render(_:)` (the only place that touches UIKit; reuses
`Formatters.byteSize`/`formatSeconds`/`formatThroughput` and the existing
`statusLabel`/`resultLabel`/`progressView`/`playerContainerView`). The VC stays
imperative (matches current style); a future TEA reducer-ification is noted, not required.

```swift
private enum ViewerState: Equatable {
    case pulling(PullProgress)                              // bytes flowing, nothing playable yet
    case preparingFirstPlayableFragment(PullProgress)      // segmenter fed; working toward first GOP/init
    case playingWhilePulling(PullProgress, ttff: Duration?)// progressive item attached & playing
    case complete(PullStats)                               // pull done; progressive still playing
    case finalizing(PullStats, source: FinalizeSource)     // durable MP4 build in flight
    case readyScrubbable(ReadyInfo)                        // current item is the durable MP4 (full scrub)
    case fallback(FallbackPhase)                           // progressive dead while pull still running
    case failed(message: String)                           // terminal failure
}
```
(`FinalizeSource = .progressiveSwap | .fallback`; `FallbackReason = .segmenterFailed |
.missingOrInvalidSegmentReport | .segmentExceedsFrozenTargetDuration |
.neverProducedFirstFrame | .progressivePlayerFailed | .truncatedReset`.) Hold state with a
guarded `didSet` that calls `render` only on real changes.

**Three concurrent concerns, three task handles** -- all created in the `@MainActor` VC, so
their closures are MainActor-isolated and every handler runs main-thread-safe; heavy work
stays off-main inside the dependencies:

- **Pull (`pullTask`)** replaces `runPull`. On `.opened`: record the URL into
  `temporaryFiles` and `startSegmenter(sourceURL:)`. Each event is fed to
  `ProgressiveAvailability.decide(event:&lastForwarded)`: `.advance(N)` forwards `N` to the
  availability stream; `.terminateProgressive` (a `.restarted`, or defensively any
  non-monotonic `.progress`) runs `handleProgressiveFailure(.truncatedReset)` -- detach the
  progressive item if attached, cancel the segmenter + server, stop forwarding availability
  -- and lets the always-on finalizer carry durable playback; `.ignore` is
  `.opened`/`.completed`. On `.completed`: finish the availability stream (a graceful EOF --
  the segmenter flushes its trailing GOP and writes ENDLIST, so a progressive item still
  attached plays to its natural end) and `onPullCompleted` (which **always** starts the
  finalizer; `segmenterTask` is hard-cancelled later, at the swap teardown, not here).
- **Segmenter (`segmenterTask`)** consumes `progressiveSegmenter.start(...)`. On
  `.opened(workDirectory:)`: track the dir for cleanup. On `.firstPlayableReady(url)`:
  `handleFirstPlayable` **first guards on state -- it attaches only while the viewer is still
  progressive-eligible (`pulling` / `preparingFirstPlayableFragment`); in any later state
  (`playingWhilePulling` / `complete` / `finalizing` / `readyScrubbable` / `fallback` /
  `failed`) it is a no-op** -- then records TTFF (`ContinuousClock` from pull start),
  `attachPlayer(url:isProgressive:true)`, `player.play()`, state -> `playingWhilePulling`. On
  throw: `handleProgressiveFailure`. The guard is load-bearing: the segmenter can emit
  `.firstPlayableReady` belatedly (slow first GOP, or its MainActor delivery interleaved after
  the finalizer has already swapped), and this ordering is **not** a progressive failure -- the
  failure path never fires, so without the guard the handler would re-attach a now-dead playlist
  URL over an already-scrubbable MP4, making the viewer *worse* than today and breaking the
  strictly-additive invariant.
- **Finalizer (`finalizerTask`)** is dispatched unconditionally on `.completed` and depends
  only on the finished `.ts`: `await clipRemuxer.remux(.ts, clipID)`, then swap (or, on the
  fallback path, attach) and tear down progressive resources.

Pure, directly-testable availability decision:

```swift
enum ProgressiveAvailability {
    enum Decision: Equatable {
        case advance(UInt64)        // forward bytes-available [0, N) to the segmenter
        case terminateProgressive   // truncation / non-monotonic progress -> kill progressive, fall back
        case ignore                 // .opened / .completed carry no availability
    }

    /// A `.restarted` (or any non-monotonic `.progress`) is terminal for the progressive
    /// attempt: the on-disk representation changed and an EVENT playlist's already-served
    /// segments cannot be rewritten, so we never re-read from 0 -- the viewer falls back.
    static func decide(_ event: ClipPullEvent, lastForwarded: inout UInt64) -> Decision {
        switch event {
        case .restarted:
            lastForwarded = 0
            return .terminateProgressive
        case .progress(let bytesWritten, _):
            defer { lastForwarded = bytesWritten }
            return bytesWritten < lastForwarded ? .terminateProgressive : .advance(bytesWritten)
        case .opened, .completed:
            return .ignore
        }
    }
}
```

`render(_:)` mapping (preserving today's strings): `pulling`/`fallback` ->
"<x> of <y>" / "<x> pulled" (today's `renderProgress`); `preparingFirstPlayableFragment`
-> "Preparing first frame"; `playingWhilePulling` -> "Playing - <progress>" + optional
"first frame <ttff>"; `complete` -> "Download complete" + the preserved
"<bytes> - <secs> - <mbps>" line; `finalizing` -> "Preparing scrubbing" (swap) or
"Preparing playback" (fallback, preserved); `readyScrubbable` -> "Ready" + the **exact**
preserved 4-part line "<pulled> pulled - <playable> playable - <secs> - <mbps>";
`failed` -> "Clip failed" + message (preserved).

### 8. Finalize + swap

On `.completed`, run the existing `ClipRemuxer.live.remux(tsURL, clipID)` to build the
durable normal MP4. Two small changes to `ClipRemuxerEngine`:

1. **Front moov:** set `writer.shouldOptimizeForNetworkUse = true` before `startWriting()`.
2. **Rebase to zero:** change `startSession(atSourceTime:)` from `firstDecodeTicks` to
   `.zero` and feed PTS/DTS rebased by `-firstDecodeTicks`, so the HLS item and the MP4
   item share origin 0 and `AVPlayer.currentTime()` maps 1:1 across the swap (the segmenter
   in section 3 rebases identically).

Swap on MainActor, preserving position and play/pause:

```swift
let resume = player.currentTime()
player.replaceCurrentItem(with: AVPlayerItem(url: mp4URL))
player.seek(to: resume, toleranceBefore: .zero, toleranceAfter: .zero) { _ in player.play() }
```

Then **cancel `segmenterTask`** and tear down the loopback server + fMP4 work dir, drop the
`.ts` (matches today's post-remux behavior), keep the `.mp4` as the viewer's tracked temp
file; state -> `readyScrubbable`. Cancelling the segmenter here is defense-in-depth behind
`handleFirstPlayable`'s state guard (section 7): the guard already neutralizes a late
`.firstPlayableReady`, and the cancel additionally stops the now-pointless pipeline and frees
its serial queue.

### 9. Fallback (strictly additive; finalize is unbreakable)

Every progressive failure routes through `handleProgressiveFailure(_:)` or the
"never produced first frame" branch and **never touches `finalizerTask`, `sourceTSURL`, or
the `.ts`**. Triggers: segmenter throws (internal error / malformed-or-unexpected TS); a
`.separable` segment arrives with a missing or non-positive `AVAssetSegmentReport` duration
(section 3 fails rather than guessing an EXTINF); a segment's reported duration would round
above the frozen `#EXT-X-TARGETDURATION` (section 4); init segment never producible (no
SPS/PPS / unsupported codec -> either throws or just never emits `firstPlayableReady` before
`.completed`); resource/player error on the progressive item (KVO `item.status == .failed`,
guarded by `currentItemIsProgressive`); optional first-frame watchdog; **truncation reset
(`.restarted`) -- terminal even if `firstPlayableReady` was already reached and the
progressive item is playing, since the append-only EVENT timeline cannot mix
representations**. On any of them: detach the progressive
player, cancel the segmenter, finish availability, delete the work dir; if the pull is
still running, go to `fallback(.awaitingCompletion(reason:))` (which renders like today's
download UI); when `.completed` arrives, the always-on finalizer produces the durable MP4
and the viewer reaches `readyScrubbable` (or `failed` only if the **finalizer itself**
fails, matching today). If progressive never reached first frame, the fallback is
literally today's path: wait for `.completed`, remux, play the finalized MP4.

  state confined to one serial domain (the writer's segment delegate may fire off-thread);
  and an explicit `ClipPullEvent.restarted` that terminates the progressive attempt on a
  200-truncation (an append-only EVENT timeline cannot mix representations), with the
  always-on finalizer still serving durable playback; and a state-guarded swap -- once the
  finalizer swaps to the durable MP4 the viewer cancels `segmenterTask` and ignores any late
  `.firstPlayableReady`, so a slow-first-GOP segmenter can never re-attach a torn-down playlist
  over a good scrubbable MP4. Alternatives considered:
  resource-loader HLS (rejected: `-12881`), single-growing-fMP4 resource loader (rejected:
  fragile for unknown length), `AVSampleBufferDisplayLayer` (rejected: no transport UI /
  surface swap; prompt requires AVPlayer), hand-rolled fMP4 box writer (escape hatch),
  single-parse-two-writers (deferred optimization). **Narrow note:** the prior problem was
  single-segment TS HLS, not localhost HLS itself -- and the deleted `LoopbackHLSServer`
  (commit `b384759`) already served cleartext loopback HLS successfully here, prior evidence
  that ATS on loopback is not a blocker.
- **Amend `app/docs/design/07-...md`** with a dated note: the "play the finalized MP4
  directly as the only path / loopback server deleted" decision is extended by ADR 08
  (progressive fMP4 adds a viewer-scoped loopback HLS server for early play; the finalized
  MP4 remains the durable artifact). Mirror the existing ADR-02 note style.
- **Amend `app/docs/design/02-...md`** clip-playback note to point at ADR 08 as well.
- **`docs/roadmap.md#Swoop lime`:** tick the progressive-playback intent on the viewer
  item; keep the scope fence (one segment per clip; no export/`tide`, timestamps/`moss`,
  incidents/`nova`, server thumbnails).

## Risks and spikes

- **AVAssetWriter segmenting on iOS 26.5** (**Phase 0 throwaway spike, gating -- run
  first**): `flushSegment()` + `expectsMediaDataInRealTime = false`; `.mpeg4AppleHLS`
  separable segments self-contained (`styp`) and playable via `#EXT-X-MAP`. Fed by the
  existing batch assembler, so it does not depend on Phases 1-3. Mitigation: hand-rolled fMP4
  box writer escape hatch, decided at the Phase 0 gate before the streaming contract is
  committed.
- **Segment-callback threading + blocking pipeline** (Phase 4/5): `AVAssetWriterDelegate` may
  fire on a thread other than the one driving `flushSegment()`, while `NWListener` serves on
  its own queue; separately the segmenter's `isReadyForMoreMediaData` spin-wait blocks its
  execution context. Mitigation: (a) copy segment bytes in the delegate and confine all
  `LoopbackMediaServer` state (playlist, path map, segment index) to one serial queue the HTTP
  handler also reads on, emitting `.firstPlayableReady`/`finish` from that queue after the
  append applies; (b) run the blocking demux -> assemble -> append pipeline on a dedicated
  serial `DispatchQueue` -- genuinely off the cooperative pool (**not an `actor`, and not a
  detached `Task`**, which is unstructured but still runs on a cooperative-pool thread; the
  finalizer's `Task.detached` is tolerable only because its one-shot spin rarely engages,
  unlike the segmenter under concurrent-preview load) -- kept distinct from the server's HTTP
  queue so a spin-wait never stalls delivery; (c) lock-guard the `failProgressive` latch (`Mutex` /
  `OSAllocatedUnfairLock`) since the delegate thread writes it and the pipeline reads it. The
  `Pelucho` integration run exercises the real delegate threads.
- **Loopback ATS** (Phase 5 spike): whether `http://localhost` needs
  `NSAllowsLocalNetworking` on iOS 26.5; `localhost` vs `127.0.0.1`. Prior evidence this is
  low-risk: the now-deleted `LoopbackHLSServer` (commit `b384759`) already served cleartext
  loopback HLS successfully in this app's own history. Mitigation: add the scoped key.
- **Swap continuity** (Phase 8 spike): `currentTime` maps cleanly across the HLS -> MP4
  swap with both timelines rebased to 0 (watch for edit-list normalization). Mitigation:
  seek with zero tolerance; if a small jump remains, accept or nudge.
- **TTFF ~= first GOP duration** (measure in **Phase 0** from the first segment's report,
  confirm in Phase 11): a segment must start on an IDR and a GOP cannot be sub-segmented, so
  the earliest first frame is bounded by the Pi's keyframe interval. **Treat this as an
  explicit go/no-go before committing Phases 5-9:** if the Pi's GOP makes early play marginal,
  the highest-leverage move is the `raspi` keyframe-interval lever (a separate side, out of
  scope here), not more app-side machinery.
- **DTS-order / one-AU-per-PES assumption**: guarded by the strict `duration > 0` throw
  in the streaming assembler (fails loud rather than mis-timing); the Pi stream is a single
  H.264 ES with no B-frames.
- **Long-clip memory**: spill fMP4 segments to a temp dir; the `.ts` and progressive work
  dir are the only growing on-disk artifacts, both viewer-scoped and cleaned up.

## Non-goals / scope fences

- **No** remuxing on the Pi; **no** AVPlayer fetch from the Pi (loopback 127.0.0.1 only);
  **no** weakening of crash-safe TS recording (Pi unchanged: raw `.ts` + `Range`).
- **No** classic faststart MP4 as the early-play mechanism (faststart is only the final
  durable artifact).
- **No** export/share UI (swoop `tide`); the finalized MP4 is produced but no save/AirDrop
  surface is added here.
- **No** multi-segment timelines / incident clips / thumbnails / timestamp provenance
  (`moss`/`nova`/deepening passes). One finished segment per clip.
- **No** cross-session clip store (`lime` clip-store item): artifacts remain per-viewer
  temp files, as today.
- **No** TEA reducer-ification of the viewer (noted as a future option; the imperative VC
  with an explicit state enum is kept).
