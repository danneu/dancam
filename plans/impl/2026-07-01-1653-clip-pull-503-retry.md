# Plan: stop collapsing every clip-serve IO error to 404 (two-sided pivot)

## Context

`serve_clip` (`raspi/service/src/clips.rs#serve_clip`) maps **every** open/stat/seek
IO error to `ClipError::NotFound` -> HTTP 404. But ADR 03's concurrency model assigns
404 a precise meaning -- "the segment is gone; it lost the race to GC, re-list or
resume by ETag" (`raspi/docs/design/03-2026-06-23-storage-ring-buffer-incident-lock.md`,
Concurrency Model). A transient `EIO`/`EBUSY`/`EACCES` on the SD card (plausible in the
hot-car / power-event environment the project treats as first-class) therefore tells the
client a present, pullable clip is permanently gone.

The review finding (`video-review.xegWVJ/05-pi-clip-serving.md` E-03) proposed a
server-only fix: return 5xx instead of 404. **That fix is ineffective on its own.** The
app's pull client treats *every* non-200/206/416 status as a terminal
`ClipPullError.http(status)` -- a `503` no differently from a `404`
(`app/DanCam/DanCam/Networking/Clips/ClipPullClient.swift#prepareBodyDecoder`, the
`guard status == 200` and the resume-`switch` `default:` arm). The viewer plays a cached
MP4 when `clipCache.lookup` hits and only pulls on a cache miss; that pull path
(`Features/ClipViewer/ClipViewerViewController.swift#runPullRemuxCacheAndPlay`) routes any
thrown `ClipPullError` through its `catch` to `fail(message:)`, which sets
`state = .failed(message:)` with no retry. Returning 503 would change the error text, not
the abandon-vs-retry behavior.

**The ideal pivot is two-sided**, so 404 keeps meaning exactly "gone" and a transient SD
hiccup is ridden out by the resume loop that already exists:

1. **Pi:** distinguish `ErrorKind::NotFound` (-> 404, genuinely gone) from other IO
   errors (-> a new 503 variant, transient).
2. **App:** treat the Pi's `503` as a *rideable retry* (`.retry(madeProgress: false)`),
   bounded by the existing consecutive-stall budget, instead of a terminal failure. 404
   and every other status stay terminal.
3. **ADRs:** record the new status-code contract as dated, append-only notes (no new
   ADRs, no supersede).

Outcome: a transient read error during the narrow pre-body open/stat/seek window now
reconnects and resumes (mirroring how mid-stream body-read drops are *already* ridden
out), while a genuine 404 still cleanly ends the pull.

## Part A -- Pi: split IO errors in `serve_clip`

File: `raspi/service/src/clips.rs` (the `ClipError` enum is private to this module; a
crate-wide grep confirms no external constructors or match sites, so the change is fully
contained here).

1. **Add a 503 variant.** Extend `ClipError` with `Unavailable`, and add the arm to its
   `IntoResponse` (the match is exhaustive with no `_`, so the compiler forces the response
   site). Mirror the existing `NotFound` simplicity -- `ClipError::Unavailable =>
   StatusCode::SERVICE_UNAVAILABLE.into_response()`, which yields an empty body
   (`Content-Length: 0` on the wire). The empty body is intentional but **not** load-bearing:
   the app decides to retry from the 503 *status line*, before constructing any body decoder
   (see Part B), so the 503 body is never read -- the connection is abandoned and the
   `Connection: close` on each request closes it. (This is deliberately a simpler shape than
   `backend.rs`'s `BackendError::CameraOffline`, which returns a `(StatusCode, &str)` tuple
   with a `"camera offline"` text body; the clip 503 carries no body.) No `Retry-After` --
   the app's backoff is client-driven.

2. **Branch the three IO sites on errno.** Replace the three
   `.map_err(|_| ClipError::NotFound)` calls (the `File::open`, `file.metadata()`, and the
   `file.seek(...)` inside the `RangeResolution::Partial` arm) with a shared helper:

   ```rust
   fn io_error_to_clip_error(error: io::Error) -> ClipError {
       match error.kind() {
           io::ErrorKind::NotFound => ClipError::NotFound,
           _ => ClipError::Unavailable,
       }
   }
   ```

   Applied as `.map_err(io_error_to_clip_error)?` at all three sites. Widen the existing
   import to `use std::{io::{self, SeekFrom}, ...}` -- the helper names `io::Error` and
   `io::ErrorKind::NotFound` *through* the `io` module, and `SeekFrom` stays bare for the
   existing `file.seek(SeekFrom::Start(...))`. Do **not** also import `ErrorKind` by name: the
   helper path-qualifies it, so a bare `ErrorKind` import is unused, and `just raspi-check`
   runs clippy with `-D warnings`. (`.kind()`/`io::ErrorKind` is used nowhere in `src/` yet.)

3. **Leave the non-IO guard as 404.** `if !metadata.is_file() { return
   Err(ClipError::NotFound); }` stays -- a directory or non-regular file is genuinely "not a
   servable clip," not a transient error. (This is also why a directory fixture stays 404;
   see tests.) The `unpullable_from` floor check at the top of `serve_clip` likewise stays
   404 (logical "not pullable," not IO).

## Part B -- App: make the Pi's 503 a rideable retry

File: `app/DanCam/DanCam/Networking/Clips/ClipPullClient.swift`.

1. **Detect `503` before the `usedRange` split.** At the top of `prepareBodyDecoder`
   (which has the only access to `head.statusCode` and exactly one call site), intercept
   exactly `503` -- the one status the Pi emits for a present-but-unreadable clip (Part A)
   -- and signal retry via a private, payload-free sentinel error:

   ```swift
   private struct RetriableServerStatus: Error {}
   ```

   ```swift
   let status = head.statusCode
   if status == 503 {
       // The Pi returns 503 only for a transient serve_clip read error (EIO/EBUSY/EACCES
       // on the SD card): the segment is present and pullable, the read just failed, so
       // reconnect and resume rather than abandon the clip. Every other status -- including
       // any other 5xx -- falls through to the terminal path below.
       throw RetriableServerStatus()
   }
   // ... existing usedRange / 200 / 206 / 416 logic unchanged ...
   ```

   Match `== 503`, not the `500...599` class: `503` is the only 5xx the Pi emits here and
   the one status RFC 9110 (section 15.6.4) defines as temporary-and-retriable. A genuine
   `500` (server bug) or a `501`/`505` (unsupported, unchanged by a retry) must surface as
   its real status, not be reconnected up to the stall ceiling and then masked as
   `.exhausted(.consecutiveStalls)`. (See the rejected-alternative note under "Non-changes.")

   Placing the check *before* the `usedRange` branch covers both the fresh-GET attempt and
   any resume attempt with one guard. Throwing before a decoder is returned is essential:
   it keeps `bodyDecoder` nil so the normal body-writing path and the tail
   "framed-complete-before-whole-file" guard never run for the `503`.

2. **Catch the sentinel in `runAttempt` -> `.retry`.** Add a clause to the body-loop catch
   chain, before the `catch let error as ClipPullError` arm. The sentinel carries no
   payload, so match the type (a bound-but-unused `let retriable as` would warn):

   ```swift
   } catch is RetriableServerStatus {
       return .retry(madeProgress: false)
   }
   ```

   At head-completion time no body bytes have been written this attempt, so
   `madeProgress: false` is correct: the reconnect counts as a no-progress stall. A
   **persistent** `503` therefore climbs to `consecutiveStalls >= maxConsecutiveStalls` and
   `producePull` throws `ClipPullError.exhausted(.consecutiveStalls)` -- bounded, not
   infinite. A **transient** `503` rides through and the pull completes. No body drain is
   needed: each attempt sends `Connection: close` and opens a fresh stream next time, and the
   Pi's 503 is `Content-Length: 0` anyway; this matches how the existing connect-failure and
   mid-stream-drop retries already abandon their stream without draining.

3. **404, other 5xx, and all other non-2xx stay terminal.** Only `503` is reclassified; the
   existing `guard status == 200` / resume-`switch` `default:` continue to throw terminal
   `ClipPullError.http(status)` for `404`, for any other 5xx (`500`/`501`/`502`/`504`/`505`),
   and for everything else. This preserves ADR 03's "404 = gone -> drop it" semantics and
   surfaces a genuine server error instead of hiding it behind a retry loop.

### App tests (`app/DanCam/DanCamTests/Networking/Clips/ClipPullClientTests.swift`)

Add a 503 wire builder beside the existing `ok200`/`partial206` helpers:
`MJPEGWireBuilder.response(statusCode: 503, headers: [("Content-Length", "0")], body: Data())`
(`MJPEGWireBuilder` already maps `503 -> "Service Unavailable"`).

- **503 then 200 completes on the fresh-GET path (`usedRange == false`):** mirror
  `retriesFromHeadWhenDropLandsBeforeAnyBody`.
  `ScriptedResponder([.finish([wire503]), .finish([ok200(total:etag:body:)])])`, then
  `requireCompleted`, `#expect(requests.count == 2)`, the on-disk
  `Data(contentsOf:) == body` check, and `#expect(result.resolvedETag == "\"1-12\"")` for the
  final 200's quoted validator -- pinning that completing *through* a 503 still captures the
  cache-key validator ADR 13 (`app/docs/design/13-2026-07-01-durable-clip-cache.md`) keys on.
  (The mirror `retriesFromHeadWhenDropLandsBeforeAnyBody` omits this assertion, so add it
  explicitly; use its `etag = "1-12"`.) (Use a clean `.finish([wire503])`, not `.drop(...)` --
  the point is that a well-framed `503`, not a transport drop, now drives the retry.) Both
  attempts here are plain GETs (`bytesWritten == 0`), so assert the retry carries no `Range:`.
- **503 on a ranged resume rides through (`usedRange == true`) -- the resume-path guard:**
  this pins that the `503` check sits *before* the `usedRange` split; an implementation that
  only handled `503` on the fresh-GET branch would pass the bullet above yet leave resume-path
  503 terminal, and this test would fail (no completion, `requests.count == 2`). Mirror
  `resumesFromLastByteAfterMidPullDrop` with a three-step responder:
  `ScriptedResponder([.drop([ok200(total: full.count, etag:, body: Data(full.prefix(5)))]),
  .finish([wire503]), .finish([partial206(start: 5, end: 11, total: full.count, etag:,
  body: Data(full.suffix(7)))])])` (a 12-byte `full`, matching the existing resume tests).
  Attempt 1 writes 5 bytes then drops; attempt 2 is the ranged resume (`Range: bytes=5-` +
  `If-Range`) answered by a clean 503 and must reconnect; attempt 3 resumes and completes.
  Assert `requireCompleted`, on-disk `== full`, `#expect(requests.count == 3)`,
  `#expect(result.resolvedETag == "\"1-12\"")` (the quoted validator established on attempt 1
  and preserved across the 503 -- the same assertion the mirror
  `resumesFromLastByteAfterMidPullDrop` already makes, and the cache key ADR 13 keys on), and
  that **both** resume requests (`requests[1]` and `requests[2]`) contain `\r\nRange: bytes=5-\r\n`
  and `If-Range:` -- proving the 503 retry kept resuming from byte 5 rather than restarting
  at 0.
- **Persistent 503 exhausts:** mirror `consecutiveStallsExhaustBudget`.
  `ScriptedResponder(Array(repeating: .finish([wire503]), count: N))` with
  `makeClient(responder, maxConsecutiveStalls: N)`, then `expectExhausted(error, reason:
  .consecutiveStalls)`, `#expect(requests.count == N)`, `#expect(tempClipFiles(clipID:).isEmpty)`.
- **404 stays terminal (boundary guard):** `ScriptedResponder([.finish([wire404])])`; assert
  the pull fails terminally with `.http(404)` and `requests.count == 1` (no retry). Add a
  small `wire404` builder and assert via `pullError`/a `.http` match (no `expectHTTP` helper
  exists in this file yet; inline the `if case .http(404)` check or add one).
- **A non-`503` 5xx stays terminal (narrow-scope guard):**
  `ScriptedResponder([.finish([wire500])])` with a small `wire500` builder
  (`MJPEGWireBuilder.response(statusCode: 500, headers: [("Content-Length", "0")], body: Data())`
  -- the builder renders an unrecognized code's reason phrase as a generic `"Status"`, which
  the head parser ignores). Assert the pull fails terminally with `.http(500)` and
  `requests.count == 1` (no retry). Companion to the 404 guard: it pins that the retry guard
  is `== 503`, not the `5xx` class, so a real server error is reported, never masked as
  `.exhausted(.consecutiveStalls)`.

## Part C -- ADRs (amend in place; no new ADRs, no supersede)

Per the root `AGENTS.md` ADR convention and the project's demonstrated practice, this pivot
*adds* a previously-unspecified contract fact and *reverses nothing* (404 keeps its meaning;
503 fills a gap; ADR 12's progress-budget thesis is untouched). So all three are dated,
append-only notes. `just adr-check` is filename-only and is satisfied trivially (no new
files).

- **`raspi/docs/design/02-2026-06-22-app-pi-transport-and-api.md`** (owns the wire contract;
  currently enumerates no clip-pull status codes): append a dated note / short "clip-pull
  error semantics" line -- `404` = NotFound / segment gone (lost the race to GC); `503` =
  transient or non-NotFound IO read error, retriable; `416` unchanged. The `503` is the only
  retriable status: the app rides exactly it as a resumable retry and treats every other
  status (including other 5xx) as terminal.
- **`raspi/docs/design/03-2026-06-23-storage-ring-buffer-incident-lock.md`** (Concurrency
  Model): append a one-line dated note clarifying that the existing "clean 404" sentence is
  specifically the GC-race / `ErrorKind::NotFound` case, and that transient read errors now
  surface as 503 (pointer to ADR 02). Decision preserved and sharpened, not changed.
- **`app/docs/design/12-2026-06-30-bounded-resilient-clip-pull.md`**: append a dated note
  extending the rideable-failure set to HTTP `503` only -- a `503` returns
  `.retry(madeProgress: false)`, counted as a no-progress reconnect against the existing
  consecutive-stall budget; all other HTTP statuses (including `404` and every other 5xx)
  stay terminal. Record *why 503 and not the 5xx class*: `503` is the Pi's deliberate
  transient-unavailable signal (RFC 9110 section 15.6.4), while a `500`/`501`/`505` is a
  genuine or permanent server failure that must surface rather than burn the stall budget.
  (`503` was terminal only by omission: ADR 12's only *reasoned* terminal HTTP case is the
  ranged-200 same-validator loop; the Pi had no 503 to ride at the time.)

## Non-changes (considered, intentionally skipped)

- **README:** no update. The change is service-internal HTTP status semantics; it touches no
  packages, `config.txt`, systemd units, Avahi/NetworkManager, or deploy paths, so the
  fresh-Pi runbook is unaffected.
- **app ADR 02 / raspi ADR 12+ / supersede:** not needed; app ADR 02 delegates the wire
  contract to raspi ADR 02 by design, and no decision is reversed.
- **Riding the whole `5xx` class (considered, rejected):** an earlier draft retried every
  `500...599` as "server-side, request was fine." Narrowed to `== 503` because that is the
  only 5xx the Pi emits for a present-but-unreadable clip (Part A) and the one status RFC
  9110 (section 15.6.4) defines as temporary-and-retriable. The other 5xx are genuine or
  permanent failures -- `500` a server bug, `501`/`505` unsupported and unchanged by a retry
  -- so riding them would spend the consecutive-stall budget and report
  `.exhausted(.consecutiveStalls)`, masking the real HTTP status a developer needs to see.
  The narrow guard keeps the app's retry contract identical to the wire contract Part C
  documents (`503` retriable, everything else terminal), with no client-only superset.
