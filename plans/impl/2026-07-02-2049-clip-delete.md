# Plan: Delete a clip (Pi `DELETE /v1/clips/{id}` + app delete UX)

## Context

The Home screen shows a read-only list of finished clips. We want the user to
delete a clip -- swipe-to-delete on the Home list plus a Delete button in the
clip viewer -- and have the footage actually removed from the Pi's SD card.

Today there is no way to remove footage from the app. This is a genuinely new
capability with two properties worth stating up front:

1. **It is the first phone-initiated mutation of already-committed footage.**
   ADR 02 (`raspi/docs/design/02-2026-06-22-app-pi-transport-and-api.md`) states
   under "Constraint reconciliation" that *every endpoint is preview/read/pull;
   nothing is on the record path*. A user clip-delete departs from that invariant
   and must be re-reconciled in a new ADR (it stays off the record path and only
   touches finished, unlocked segments, so it is defensible -- but the written
   contract does not currently contemplate it).

2. **It is net-new roadmap scope.** Clip deletion is not a named swoop or an
   Icebox item. `kelp` (SD management) is card-level (format), `nova` (incident
   lock) is the *inverse* (protect footage), and `lime` (watch clips) is read-only.

### Design decisions (resolved with the user)

- **Reconciliation: emit a symmetric `clip_removed` SSE event.** The app removes
  the row optimistically on a `2xx`, and the Pi also pushes a `clip_removed` delta
  on `/v1/events` (mirror of the existing `clip_finalized`). This keeps the events
  plane symmetric, reconciles every connected client, and gives future GC eviction
  a list-removal channel for free.
- **Confirmation: a destructive confirm alert** precedes the DELETE (irreversible;
  the SD card has no trash/undo).
- **Surface: Home swipe-to-delete AND a Delete button in the clip viewer.** The
  viewer must be given the store (today it receives only `dependencies` + `clip`).
- **Storage: delete is a new mutation on the existing `StorageCoordinator`.** The
  single-writer coordinator now exists
  (`raspi/service/src/storage.rs#StorageCoordinator`, ADR 16
  `raspi/docs/design/16-2026-07-02-storage-coordinator-segment-id-witness.md`).
  It already owns start-segment allocation under one mutation mutex and persists a
  durable `state/state.json` `high_water_seq` witness so a deleted id can never be
  reissued. Clip delete lands as `StorageCoordinator::delete_finished_segment`,
  the write-ahead-delete mutation ADR 16 explicitly reserved a seam for: raise
  `high_water_seq >= id` durably *before* unlinking, then remove every on-disk path
  for the id and fsync the recording dir. The handler calls the coordinator; it does
  not own the durable unlink. This is the ideal shape ADR 16 anticipated -- no interim
  in-process lock, no second witness file, no re-implementation of allocation.

---

## Part A -- Pi service (Rust, `raspi/service/`)

### A0. Coordinator-owned delete + start/delete ordering (storage)

The coordinator (`storage.rs#StorageCoordinator`) already holds `rec_dir` and a
`mutation: Mutex<()>` and exposes `allocate_start_segment` under that mutex, which
reads the `state/state.json` `high_water_seq` witness (`storage.rs#read_witness`),
maxes it with `clips::max_clip_seq`, persists `next` (`storage.rs#persist_witness`,
temp -> fsync -> rename -> dir-fsync), and returns `next`. ADR 16 records the
standing rule for any future unlink: **"any future mutation that removes a segment
file must first persist `high_water_seq >= seq` under the same coordinator mutex."**
Rollover ids are minted outside the coordinator and are *not* written to the witness
(they exceed it and are covered only by the directory scan while their files exist),
so a delete of the top finished clip -- which may be a rollover id above the persisted
witness -- must write-ahead the witness itself, or the next allocation would drop back
to `max_clip_seq` and reissue the id. Every "id alone is the stable identity"
assumption in this plan (event ids, optimistic-remove keys, the absence of `If-Match`,
the harmless stale `DurationCache` entry) rests on that write-ahead. Delete is built
here, not deferred.

#### A0.1 The write-ahead-delete mutation `delete_finished_segment`

New method on `StorageCoordinator`, taking the id and a floor reader (A0.2 explains
the closure):

```rust
pub fn delete_finished_segment(
    &self,
    id: SegmentId,
    live_floor: impl FnOnce() -> Option<SegmentId>,
) -> Result<(), SegmentDeleteError> {
    let _guard = self.mutation.lock().expect("storage coordinator mutex poisoned");
    let rec_dir = self.rec_dir.as_ref();

    // (1) Floor check at the coordinator boundary, INSIDE the mutex (A0.2). A
    //     concurrent start's reservation commits its floor under this same mutex,
    //     so a delete can never remove an active or newly reserved segment.
    if live_floor().is_some_and(|floor| id >= floor) {
        return Err(SegmentDeleteError::NotFound);
    }

    // (2) Resolve the clip by scanning rec_dir for EVERY regular file whose parsed
    //     seq == id (bare and/or stamped duplicates). No dedupe, no reconstructed
    //     segment_filename(id) (A0 rationale below).
    let paths = clips::segment_paths_for_id(rec_dir, id).map_err(SegmentDeleteError::Io)?;
    if paths.is_empty() {
        return Err(SegmentDeleteError::NotFound);       // -> 404
    }

    // (3) Write-ahead: raise high_water_seq to at least `id` and make it durable
    //     BEFORE the first unlink. read_witness fails closed on a corrupt witness.
    let existing = read_witness(rec_dir).map_err(SegmentDeleteError::Io)?.unwrap_or(0);
    persist_witness(rec_dir, existing.max(id)).map_err(SegmentDeleteError::Io)?;

    // (4) Remove every matched path, then (5) fsync the recording dir so the
    //     unlinks are durable.
    for path in &paths {
        std::fs::remove_file(path).map_err(SegmentDeleteError::Io)?;
    }
    fsync_dir(rec_dir).map_err(SegmentDeleteError::Io)?;
    Ok(())
}
```

- **Scan, do not reconstruct.** New helper `clips::segment_paths_for_id(rec_dir, id)
  -> io::Result<Vec<PathBuf>>`, `pub(crate)`, next to `segment_candidates`: read
  `rec_dir`, keep entries whose `parse_segment_filename` yields `seq == id` and whose
  `metadata.is_file()`, and return **all** such paths. It is deliberately *not*
  `segment_candidates` (which applies `dedupe_candidates`): a live/just-finished
  segment is written under a *stamped* name (`recorder::stamped_segment_filename`),
  and a bare `seg_{id:05}.ts` plus a stamped name for the same id can coexist
  (`clips.rs#dedupe_candidates` collapses that duplicate only in *listings*).
  Reconstructing the bare `recorder::segment_filename(id)` path would `404` a valid
  stamped clip; deduping would unlink one path and leave a ghost the list still shows.
  A `read_dir` failure (e.g. an unreadable rec_dir) returns `Err` -> 503; a directory
  named like a clip is not `is_file()`, so it is not collected (an id with only such
  an entry -> empty -> `NotFound` -> 404, matching `serve_clip`'s guard).
- **Write-ahead witness raise.** Reuse the coordinator's own `read_witness` /
  `persist_witness` / `state_path` / `fsync_dir` -- the same `state/state.json`
  `high_water_seq` file and atomic temp->fsync->rename->dir-fsync protocol
  allocation uses. Raise to `existing.max(id)` so it never lowers and always ends
  `>= id`. A **present-but-corrupt** witness makes `read_witness` return
  `Err(InvalidData)` -> `SegmentDeleteError::Io` -> 503 with *nothing unlinked* --
  never silently treated as `0`, which could later reissue a deleted id. Note the status
  asymmetry with allocation, which surfaces the *same* corrupt witness as a `500`
  (`BackendError::Storage`): the delete path deliberately keeps this persistent condition on
  `503` (ADR 17 records why -- delete's `503` is broader than the read path's transient-only
  `503`).
- **Error type.** New `SegmentDeleteError { NotFound, Io(io::Error) }`. The handler
  (A1) maps `NotFound` -> 404 and `Io` -> 503. Distinguishing the two (rather than an
  `io::Result` where a post-unlink `NotFound` kind would wrongly become a 404) is what
  lets the layered contract (A0.3) hold: only the pre-effect "no matching file" and
  the floor block are `NotFound`; every failure at or after the witness raise is `Io`.

#### A0.2 Structural floor-before-file ordering: the reservation turn

`allocate_start_segment` releases the coordinator mutex *before* the backend drives
`Input::StartCommand`, and `Input::StartCommand` is what sets the recorder floor
(`recorder.rs#RecorderState::start` sets `unpullable_floor = Some(start_segment)`;
surfaced as `backend.unpullable_from()`). There is a witness->floor window: a start has
allocated (and persisted the witness for) id `N` but the floor is not yet `N`.

**In production this window is not exploitable**, because in *both* backends the segment file
for `N` is created strictly *after* the floor is set. The mock spawns
`run_mock_recording_writer` (which opens `seg_N`) only after `drive_now(Input::StartCommand)`
(`backend.rs#MockRecorder::start`), and the camera drives `Input::StartCommand` before it issues
`ChildCommand::StartRecording` to the child that writes the file
(`camera/mod.rs#CameraBackend::start_recording`). So during the witness->floor window `seg_N`
does not exist yet: a concurrent `delete_finished_segment(N)` scans, finds nothing, and `404`s
with nothing unlinked; and once the file does exist the floor is already `Some(N)`, so the
in-mutex floor check refuses `N >= N`. The runtime guarantee against unlinking an active segment
is therefore the in-mutex floor check **plus** this file-after-floor ordering -- not a lock
spanning allocation, and not something delete has to race.

What the reservation turn adds is making that ordering *structural* rather than incidental, so a
future start path cannot regress it. An earlier draft reached for a `Backend::segment_id_lock`
held from before allocation through `StartCommand`. With the coordinator in place that lock is
redundant and wrong-shaped: the coordinator mutex is already the single-writer seam.
**Order the floor commit inside the same coordinator turn as allocation**, so delete's
in-mutex floor check is strictly before or after a start's reservation, never between:

```rust
/// Allocate the next start segment and let the caller commit the reservation (drive
/// Input::StartCommand, which sets the recorder floor) while the mutation mutex is
/// still held, so a concurrent delete_finished_segment floor check cannot interleave
/// between the witness persist and the floor becoming visible.
pub fn reserve_start_segment<R>(
    &self,
    commit: impl FnOnce(SegmentId) -> R,
) -> io::Result<(SegmentId, R)> {
    let _guard = self.mutation.lock().expect("storage coordinator mutex poisoned");
    let next = /* read_witness + max_clip_seq + max + saturating_add(1), as today */;
    persist_witness(self.rec_dir.as_ref(), next)?;    // witness durable first
    let committed = commit(next);                     // floor commit, still under the mutex
    Ok((next, committed))
}
```

- `reserve_start_segment` becomes the **only** start-allocation entry point on the
  production surface -- there is no public no-floor allocator a backend could reach for by
  mistake, so a start path that failed to commit the floor cannot compile (the only way to
  obtain an id is to pass a `commit`). This is what enforces the reservation turn on
  production starts rather than leaving it to convention: it is the structural expression of
  the file-after-floor invariant above -- since the only way to obtain an id is to commit the
  floor in the same mutex turn, no future start path can reorder file-creation ahead of the
  floor and reopen the witness->floor window. Defense-in-depth, not the closing of a
  currently-reachable hole. The existing no-floor
  `allocate_start_segment` survives **only** as a `#[cfg(test)]` inherent-method wrapper
  (`self.reserve_start_segment(|_| ()).map(|(id, _)| id)`), so the many same-crate
  `storage.rs` allocation unit tests keep their readable call sites; its behavior and every
  existing `storage.rs` allocation assertion are unchanged. Integration tests in
  `tests/clips.rs` are a separate crate and cannot see a `#[cfg(test)]` method, so where
  they need a bare id they call the `reserve_start_segment(|_| ())` form inline.
- `commit` runs synchronously under the `std::sync::Mutex` (no await); `drive_now`
  is synchronous, so nothing is held across an await. Lock order is always
  coordinator-mutex -> `EventHub` inner mutex in both the reservation commit and the
  delete floor read, so there is no deadlock with independent hub locks
  (`connect`/`tick`/telemetry never take the coordinator mutex).
- **Backend wiring.** Both starts swap `allocate_start_segment()` for
  `reserve_start_segment(commit)`, after their existing guards (so a start that
  no-ops into an already-running writer still never touches the witness). The commit
  closure is exactly the `Input::StartCommand` drive they already do; the returned
  events feed `starting_session` as before -- no new await, no reordering of the
  writer spawn / child command that follows.
  - `backend.rs#MockRecorder::start`:
    ```rust
    let (start_segment, events) = self
        .storage
        .reserve_start_segment(|seg| self.hub.drive_now(Input::StartCommand { start_segment: seg }))
        .map_err(|error| { tracing::error!(%error, "start segment allocation failed"); BackendError::Storage })?;
    let Some(session) = starting_session(&events) else { return Ok(()); };
    ```
  - `camera/mod.rs#CameraBackend::start_recording`: identical shape; `session` and
    `start_segment` then feed `ChildCommand::StartRecording` as today.

#### A0.3 Layered failure contract and the id-non-reuse invariant

`delete_finished_segment` is not all-or-nothing on the filesystem -- unlinking a
bare+stamped duplicate is two `remove_file`s plus a dir-`fsync`, none atomic as a set
-- so the honest guarantee is layered, and the handler emits `clip_removed` + `204`
**only after** the mutation returns `Ok`:

- **Failure before any unlink** (floor block, scan error, or the witness raise) returns
  `404`/`503` having changed nothing on disk -- genuinely no-effect.
- **Failure once unlinking has begun** (a later duplicate `remove_file`, or the final
  dir-`fsync`) still returns `503` with no `clip_removed`, but one or more paths for
  the id may already be gone. Past that point the guarantee narrows to *not durably
  acknowledged, no event emitted* -- not *no filesystem effect*.
- **The safety invariant holds unconditionally**: the witness is raised and fsync'd
  *before* the first unlink, so `high_water_seq >= id` survives a partial or failed
  delete and a deleted id is never reissued. Byte-for-byte atomicity of the unlink set
  is not the property that matters; no id reuse is.
- **A partial delete self-heals.** The app re-inserts the row on the `503` (B4); a
  surviving path keeps the clip resolvable (`resolve_segment` re-scans on every call),
  so a **repeat DELETE** finishes the unlink (`204`, or `404` once nothing remains),
  and a **head reload** prunes the row if the fresh listing no longer shows it
  (`reconciledHead`, B4).

This honors ADR 03's storage discipline ("every directory mutation is followed by an
`fsync()` of the containing directory") that ADR 16's coordinator formalizes: a power
loss after the `204` can neither resurrect the clip (unlink was fsync'd) nor reissue
its id (witness was fsync'd first). Delete does **not** touch the record path: it
never runs while a segment for that id is open (floor guard), and start-allocation is
unchanged -- the witness is still written only at reservation and now at delete, never
while recording rolls segments.

### A1. The `DELETE /v1/clips/{id}` handler

**Route** -- `raspi/service/src/lib.rs#app`: chain the verb onto the existing path:

```rust
.route("/v1/clips/{id}", get(clips::serve_clip).delete(clips::delete_clip))
```

The three existing middleware layers (host allowlist -> `421`, proto headers,
request trace) wrap it automatically.

**Handler** -- new `clips.rs#delete_clip`. It validates the mutation headers, then
calls the coordinator (the handler owns no durable unlink logic) via `spawn_blocking`
(the mutation does sync fs + fsync, exactly like `serve_clip` / `list_clips` push
their fs work off the runtime):

```rust
pub async fn delete_clip(
    State(state): State<AppState>,
    PathParam(id): PathParam<u32>,
    headers: HeaderMap,
) -> Result<StatusCode, DeleteClipError> {
    require_mutation_headers(&headers)?;               // 415 / 400 (shared gate, A3)

    let storage = state.storage.clone();
    let backend = state.backend.clone();
    tokio::task::spawn_blocking(move || {
        // The coordinator takes the mutation mutex, checks the floor INSIDE it via
        // this closure (ordered against a start's reservation, A0.2), scans for every
        // path for `id`, write-ahead-raises the witness, unlinks, and dir-fsyncs.
        storage.delete_finished_segment(id, || backend.unpullable_from())
    })
    .await
    .map_err(|_| ClipError::Unavailable)?              // join failure -> 503
    .map_err(segment_delete_to_clip_error)?;           // NotFound -> 404, Io -> 503

    // Emit + acknowledge ONLY after the delete is durable, so a power loss after the
    // 204 can neither resurrect the clip nor let its id be reused.
    state.backend.note_clip_removed(id);               // emit clip_removed (A2)
    Ok(StatusCode::NO_CONTENT)                          // 204
}
```

Reuse verbatim, do not reinvent:
- id -> path(s) and the floor: both live in the coordinator now
  (`delete_finished_segment`, A0). The handler passes the floor reader
  `|| state.backend.unpullable_from()` (`clips.rs#serve_clip` uses the same
  `backend.unpullable_from()`), evaluated *inside* the coordinator mutex so a
  concurrent start cannot reserve the id between the check and the unlink.
- error mapping: `segment_delete_to_clip_error` maps
  `SegmentDeleteError::NotFound -> ClipError::NotFound` and
  `SegmentDeleteError::Io(_) -> ClipError::Unavailable`, reusing `clips.rs#ClipError`'s
  existing `IntoResponse` (404 / 503) -- the same status contract `serve_clip` uses.

**Idempotency / repeat-safety semantics** (mirror the clip-pull `404` note in ADR
02): a DELETE of an already-gone or at/above-floor clip returns `404`, not an error --
the app treats `404` as "already removed, keep it removed" (see B4). A successful,
durable delete returns `204` and is the *only* path that emits `clip_removed`; the
event fires exactly once, only after the removal has been fsync'd, on the
present->absent transition. A DELETE that failed *mid-unlink* (a `503`, A0.3) is safe
to retry: the retry re-scans and either unlinks the surviving path(s) for a clean
`204` or returns `404` once none remain, so a partial delete always converges (or the
next head reload prunes it).

**`DeleteClipError`**: small wrapper so the handler can return either a mutation-gate
error or a `ClipError`. Give it `IntoResponse` delegating to each (or add
`From<ClipError>` and `From<MutationHeaderError>` so the two `?` sites convert).

> Optional cleanup, not required: ADR 16's `delete_finished_segment` doc-note also
> anticipates invalidating the id's cached duration. `ts_duration::DurationCache` has
> no `invalidate` today. A deleted seq's cache entry lingers but is harmless -- ids
> are never reused (A0), so it can never be queried again. If a
> `DurationCache::invalidate(id)` is trivially cheap, call it best-effort from the
> handler after `Ok`; otherwise skip.

### A2. The `clip_removed` event (contract + Rust + wiring)

**Wire contract fixture** -- new `contract/events/clip_removed.json`:

```json
{ "type": "clip_removed", "id": 42 }
```

`id` is the segment `seq` -- the stable clip identity already used by
`clip_finalized.id`, `/v1/clips[].id`, and the app's `Clip.id`. `id` alone is
sufficient (ids are never reused -- A0). Update `contract/events/README.md`'s event list.

**Rust event enum** -- `raspi/service/src/events.rs#Event`: add

```rust
ClipRemoved { id: u32 },
```

and extend the golden-corpus test trio in the same file: `canonical_events()` (add
`Event::ClipRemoved { id: 42 }`), `canonical_name()` (`=> "clip_removed"`). The
existing `events_match_the_golden_corpus` test then asserts serialize == fixture and
decode == value automatically.

**World input** -- `raspi/service/src/world.rs`: add `Input::ClipRemoved { id: u32 }`
and handle it in `World::apply` as a pure emit (it does not touch recorder state --
the clips list is filesystem-derived, not part of `World`):

```rust
Input::ClipRemoved { id } => vec![Event::ClipRemoved { id }],
```

**Backend hook** -- `raspi/service/src/backend.rs#Backend`: add `note_clip_removed` as a
**required** trait method (no default body), unlike the optional `tick` /
`update_telemetry` hooks:

```rust
fn note_clip_removed(&self, id: SegmentId);
```

A default no-op would let a real backend that forgot the override still compile and still
return `204`, silently dropping the client-visible `clip_removed` reconcile frame -- the
one behavior the delete's success path exists to produce. Making the method required turns
that silent correctness bug into a compile error, so every `Backend` must state its
behavior. (No `segment_id_lock` accessor is added -- the coordinator mutex is the
serialization seam now, reached through `state.storage`, not the backend.)

Implement it at all five `impl Backend` sites (grep `impl Backend for`):
- `backend.rs#MockBackend` and `camera/mod.rs#CameraBackend` (both hold `hub: Arc<EventHub>`)
  drive the hub: `fn note_clip_removed(&self, id: SegmentId) { self.hub.drive_now(Input::ClipRemoved { id }); }`.
- `tests/clips.rs#StubBackend` drives its hub the same way -- A5's event-emission case
  subscribes for the delta, so this stub must really emit.
- `tests/preview.rs#StubBackend` and `tests/status.rs#StubBackend` add an explicit
  `fn note_clip_removed(&self, _id: SegmentId) {}`. These suites do not assert on events,
  but writing the empty body out is exactly the point: the compiler now forces every
  backend to declare its behavior rather than inherit silence.

`EventHub::drive_now` already bumps `seq`, broadcasts to `/v1/events`, and is the
same path `clip_finalized` flows through -- no new plumbing.

### A3. Shared mutation-header gate

`recording.rs#require_mutation_headers` (Content-Type: application/json -> 415;
non-empty `Idempotency-Key` -> 400) is defined in `recording.rs` and returns a
`recording`-specific `RecordingRequestError`. It already has **two** callers today --
`recording.rs`'s start/stop handlers and `time_sync.rs#sync_time` (which imports
`recording::{require_mutation_headers, RecordingRequestError}` and wraps it in
`TimeSyncError::MutationHeaders`) -- and `delete_clip` (A1) makes three. Extract it into a
neutral module so every mutating endpoint reuses the identical CSRF/mutation contract (ADR
02: "every mutating endpoint requires `Content-Type: application/json` and
`Idempotency-Key`") without coupling to `recording`. All three call sites must be updated in
this change -- moving the fn while touching only `recording.rs` would leave `time_sync.rs`
importing a symbol that no longer exists.

- New module `raspi/service/src/mutation.rs` (declared `mod mutation;` in `lib.rs`)
  holding `pub(crate) fn require_mutation_headers` returning a new `MutationHeaderError`
  enum with `IntoResponse` (415 / 400) and its unit tests (move the three existing header
  tests from `recording.rs`).
- `recording.rs`: delete its private copy; call the shared fn. `RecordingRequestError`
  wraps `MutationHeaderError` (or is folded into it) -- whichever is cleaner -- with a
  `From<MutationHeaderError>` so the existing `?` at both handler sites still converts.
- `time_sync.rs`: switch the import from `recording` to `mutation`; change
  `TimeSyncError::MutationHeaders` to wrap `MutationHeaderError` (not `RecordingRequestError`)
  and update its `From` impl, so `sync_time`'s existing `require_mutation_headers(&headers)?`
  and the 415/400 `IntoResponse` mapping are unchanged in behavior. Do not leave it importing
  a moved symbol from `recording` or wrapping a `recording`-specific error type.
- `delete_clip` (A1): returns `MutationHeaderError` through its `DeleteClipError` wrapper --
  the third caller of the shared gate.

### A4. Fail-closed `/v1/clips` list (precondition for B4's authoritative head reconciliation)

B4 makes a head response authoritative for every id `>= lower`: an id the head omits is
pruned and tombstoned for the session. That is only safe if an empty-or-short head means
"these clips are really gone," never "the Pi could not read them." Today `/v1/clips` fails
**open** in two spots, so a transient fault masquerades as a truthful deletion:

- `clips.rs#segment_candidates` swallows a scan error --
  `let Ok(entries) = std::fs::read_dir(rec_dir) else { return Vec::new(); }` -- so an
  unreadable *existing* rec dir yields an empty list.
- `clips.rs#list_clips` maps a `spawn_blocking` **join** failure to `(Vec::new(), None)` and
  still returns `200`, so a panicking list task yields an empty list.

Either would drive B4 to prune every clip and plant permanent in-session tombstones, hiding
footage that still exists on the card. Make the list path distinguish "empty" from "couldn't
read" and fail closed:

- Thread the scan error out instead of swallowing it. The shared scan primitive
  `segment_candidates` returns `io::Result<Vec<SegmentCandidate>>`: a `read_dir`
  `ErrorKind::NotFound` stays the intentional empty (a fresh Pi with no rec dir yet --
  preserve `read_finished_clips_returns_empty_for_missing_dir`), but any other `read_dir` /
  `metadata` error propagates as `Err`. **Every transitive caller must then adapt** -- the
  signature change does not compile until all of them are handled, and each makes an explicit
  fail-open-vs-closed decision at its own boundary (do not let `serve_clip`/enrich stay
  fail-open by accident):
  - `read_finished_clips` -> `io::Result<(Vec<ClipMeta>, Option<SegmentId>)>` -- **fail closed**;
    `list_clips` maps `Err` to `503` (below). (Missing dir still yields the empty page.)
  - `open_segment` -> `io::Result<Option<(u32, PathBuf, u64)>>` and its caller `max_clip_seq` ->
    `io::Result<Option<u32>>` -- **fail closed**; the coordinator allocation scan `?`-propagates
    (start `500`, below). (Missing dir still yields `None`.)
  - `resolve_segment` -> `io::Result<Option<SegmentCandidate>>`, and its three consumers each
    decide:
    - `serve_clip` (`clips.rs`): map the scan `Err` to `ClipError::Unavailable` -> **`503`**, keep
      `Ok(None)` -> `404`. This is a deliberate behavior change: today an unreadable rec dir makes
      `resolve_segment` return `None` -> `404` (fail-open), which both contradicts ADR 02's "`503`
      = an IO failure prevented serving a present clip" and is internally inconsistent with
      `serve_clip`'s existing file-permission-denied -> `503`
      (`serve_clip_returns_unavailable_for_permission_denied_file`). Fail closed removes that
      latent inconsistency. (A directory named like a clip and a missing id still `404`: the
      `read_dir` *succeeds*, the entry is just filtered / not found.)
    - `clip_meta` -> `io::Result<Option<ClipMeta>>`: `finalized_clip_meta` (both backends) already
      folds a `spawn_blocking` join failure to `None`, which the recorder treats as fatal
      (`Input::Fail` on the rollover/stop path). Fold the scan `Err` the same way, so a broken rec
      dir during finalize stays that existing fatal-fail rather than a silent success -- **fail
      closed**, as a recorder fail rather than an HTTP status.
    - `enrich_current_segment` (`events.rs`): this only decorates the live snapshot's
      `current_segment` with a duration and is advisory. On a scan `Err`, **skip** enrichment
      (leave `dur_ms = None`) rather than `503`-ing the whole `/v1/status` + SSE snapshot -- a soft
      degrade, the one caller that intentionally stays fail-open.
- `list_clips` maps the scan `Err` **and** the `spawn_blocking` join failure to
  `ClipError::Unavailable` -> `503` (reuse `clips.rs#io_error_to_clip_error` / `ClipError`'s
  existing `IntoResponse`), not an empty `200`. Its return type moves from
  `Result<Json<ClipsResponse>, StatusCode>` to `Result<Json<ClipsResponse>, ClipError>`, so
  `503` is shared with `serve_clip` and delete; the existing `400` arms stay (map to
  `ClipError` or keep a small wrapper -- whichever is cleaner).
- The coordinator's allocation scan inherits the same fail-closed behavior for free and
  strictly safer: `reserve_start_segment`/`allocate_start_segment` already return
  `io::Result`, so a `max_clip_seq` scan error now propagates (`?`) instead of silently
  reading as "no clips" and lowering the id floor -- the same fail-closed stance ADR 16 takes
  for a corrupt witness. (This touches accepted allocation code; it is a strict hardening, not
  a behavior change on the happy path, and the ADR 17 record notes it.)

This is a genuine prerequisite for B4, not scope creep: the app may treat absence as a delete
only if the Pi guarantees absence is truthful.

### B4. `ClipsFeature` actions (optimistic delete + reconcile + stale-input suppression)

`Features/Clips/ClipsFeature.swift`. The list is rebuilt by `merged(...)` from several
inputs -- `.clipFinalized`, the `.load`/`.refresh` head response, and paged responses. A
delete that only does `removeAll` is not durable: a `/v1/clips` read already in flight
when the user deletes still carries the row, and `merged` would re-insert it after the
optimistic removal (and possibly after the `clip_removed` SSE already reconciled). So
State carries a suppression set that `merged` filters against.

State (`pendingDeleteIDs`, `suppressedClipIDs`, and `clipFinalizeEpoch` are new; `headEpoch`
**replaces the existing `loadEpoch`** -- see the note below):
```swift
var pendingDeleteIDs: Set<Int> = []      // ids with a local DELETE in flight, eligible for failure re-insert
var suppressedClipIDs: Set<Int> = []     // authoritative tombstones: deleted locally, via clip_removed, or head-pruned
var headEpoch = 0                        // was `loadEpoch`: head-load generation, now bumped when a head load (.load/.refresh) LAUNCHES
var clipFinalizeEpoch: [Int: Int] = [:]  // clip id -> headEpoch at the time it was folded in
```
`headEpoch` is the existing `loadEpoch` field renamed and retimed, not a second counter. Today
`loadEpoch` bumps when a `.clipsResponse` *succeeds* (response-time) and gates stale pages
(`pageResponse` guards `epoch == state.loadEpoch`; `loadMore` captures it). B4 moves the bump to
`.load`/`.refresh` *launch* (request-time) so a clip finalized while a head request is in flight
can be stamped with that request's epoch; `loadMore` still captures it and `pageResponse` still
guards on it, so page invalidation is unchanged for every production reset path -- a head reset
still runs through a `.refresh`/`.load`, which now bumps at launch, so an in-flight page still
straddles the advance and is dropped. One behavioral divergence at the margin: a page whose request
*launches during* an in-flight refresh now captures the already-bumped epoch and is **kept** rather
than dropped (under response-time it captured the pre-bump epoch and the head's completion
invalidated it). This is harmless because every `merged` call is suppression-filtered -- a kept
stale page can only re-add ids the authoritative head did not tombstone, never resurrect a
deleted-or-pruned clip. Renaming it `headEpoch` keeps the two epoch roles legible; there is no
leftover `loadEpoch`.

Actions (mirroring `RecordingFeature`'s `Result<Bool, Error>` trick to stay `Equatable`):
```swift
case deleteTapped(Clip)
case deleteResponse(clip: Clip, Result<Bool, ClipsError>)
case clipRemoved(id: Int)     // folded from the SSE event
```
The existing head-load action also gains a request token so the reducer can tell a
finalize-in-flight from a stale phantom: `case clipsResponse(Result<...>)` becomes
`case clipsResponse(epoch: Int, Result<ClipsResponse, ClipsError>)`, mirroring the existing
`pageResponse(epoch:, ...)`. `.load`/`.refresh` bump `state.headEpoch` and thread it
through `fetchEffect`, so the completing `.clipsResponse` carries the epoch of the request
that produced it.

Reducer:
- `.deleteTapped(clip)`: `state.pendingDeleteIDs.insert(clip.id)` **and**
  `state.suppressedClipIDs.insert(clip.id)`; optimistic remove
  `state.clips.removeAll { $0.id == clip.id }`; return
  `.run(id: "clip-delete-\(clip.id)")` calling `dependencies.clips.delete(clip.id)`,
  dispatching `.deleteResponse(clip:, .success(true))` / `.failure(error)` (same
  cancellation handling as `fetchResult`).
- `.deleteResponse(clip, .success)`: `state.pendingDeleteIDs.remove(clip.id)` -- the delete
  resolved; the row stays removed and suppressed (the `clip_removed` SSE also arrives and
  is idempotent).
- `.deleteResponse(clip, .failure(.http(404)))`: `state.pendingDeleteIDs.remove(clip.id)`
  -- already gone; stays removed and suppressed. (404 is not a re-insert condition.)
- `.deleteResponse(clip, .failure(let error))` (any other): re-insert **only if the id is
  still pending**. `guard state.pendingDeleteIDs.contains(clip.id)`:
  - **still pending** -> the local delete is the only authority that touched this id, and it
    failed: `state.pendingDeleteIDs.remove(clip.id)`, `state.suppressedClipIDs.remove(clip.id)`,
    `state.clips = merged(existing: state.clips, incoming: [clip], suppressed: state.suppressedClipIDs)`,
    and `state.status = .failed(error.displayMessage)` so the failure surfaces.
  - **not pending** -> an authoritative tombstone (`.clipRemoved` or a head prune, below)
    already removed this id while the delete was in flight. Drop any stale pending marker and
    do nothing else: do **not** un-suppress, do **not** re-insert, do **not** surface an error.
    The clip is legitimately gone; resurrecting it would undo a confirmed removal. This is
    the fix for "a late delete failure undoes an authoritative removal."
- `.clipRemoved(id)`: `state.suppressedClipIDs.insert(id)`;
  `state.pendingDeleteIDs.remove(id)` (an authoritative removal supersedes any in-flight
  local delete of the same id, so that delete's later failure can no longer re-insert it);
  `state.clips.removeAll { $0.id == id }` (idempotent; also covers a delete initiated by
  another client or future GC).

`merged` gains a `suppressed: Set<Int>` parameter and drops any id in it from the final
result; **every** call site passes `state.suppressedClipIDs` (`.clipFinalized`,
`.pageResponse(epoch:, .success)`, the re-insert path above, and -- via `reconciledHead`
below -- `.clipsResponse(epoch:, .success)`). The two sets play distinct roles:
`suppressedClipIDs` is the authoritative "this id is gone, filter it from any merge"
tombstone (set by `.deleteTapped`, `.clipRemoved`, and head-prune); `pendingDeleteIDs` is
the narrow "a local DELETE for this id is in flight and its failure may re-insert" marker
(set only by `.deleteTapped`; cleared by every `.deleteResponse` and by any authoritative
tombstone). Because ids are never reused (Pi A0), a suppressed entry can never be
legitimately re-created, so the tombstone set is removed-from *only* by a still-pending
failure re-insert; its size is bounded by the number of removals observed in a session
(negligible).

**Head-load reconciliation (missed-delete backstop).** Suppression only covers ids the
app *itself* deleted or saw a `clip_removed` for. If a `clip_removed` is missed across a
reconnect gap, or a delete succeeds server-side but the app sees only a transport failure
(so the failure path above re-inserts the row), the phantom row would otherwise live
forever, because `merged` is a union and never drops an existing id absent from an
incoming list. So the head load is made **authoritative for every id it could have
enumerated** -- from its `next_cursor` boundary *upward with no upper bound* --
`.clipsResponse(epoch:, .success)` routes through a new
`reconciledHead(existing:pending:response:suppressed:requestEpoch:finalizeEpoch:)` instead of
plain `merged`:

- `lower = Int(response.nextCursor) ?? 0` -- the oldest id this page enumerated.
  `next_cursor` is **inclusive**: the Pi returns it as the oldest id *in* the page and the
  next page is `seq < next_cursor` (verified in `clips.rs#read_finished_clips` and its
  `read_finished_clips_pages_newest_first_with_boundaries` test: a `[4,3]` page yields
  `next_cursor == 3`). `nil` means the list drained to its oldest, so the head is
  authoritative all the way down (`lower == 0`).
- The **authoritative-absent set** is every id `>= lower` that is absent from
  `response.clips` and is *not* finalize-in-flight (next bullet), drawn from the union of
  `state.clips` **and `pendingDeleteIDs`** -- not just the current list. Including the pending
  set is load-bearing: `.deleteTapped(7)` optimistically removes 7 from `state.clips`, so if
  the universe were only the current list, a head that omits 7 could never confirm that
  delete, and a later `.deleteResponse(7, .failure(non-404))` would still find 7 pending and
  re-insert it. Every id in the authoritative-absent set is recorded as a tombstone: pruned
  from `state.clips` if present, inserted into `suppressedClipIDs`, and **removed from
  `pendingDeleteIDs`**. Clearing pending is what stops that late failure from re-inserting a
  row the head already confirmed gone, and a stale in-flight page from re-adding it.
- The one carve-out: an absent id is **kept** iff `finalizeEpoch[id] >= requestEpoch` --
  i.e. it was folded in by a `.clipFinalized` that arrived *after this head request
  launched*, so the server legitimately built its response before that clip existed. This
  is the precise replacement for the prior round's id-position heuristic ("keep everything
  above the newest returned id"), which could not tell a real finalize-in-flight from a
  stale clip deleted elsewhere whose id merely happens to be higher than the current
  server max.
- An existing clip with `id < lower` is older than this page and owned by paging -> keep.
- An **empty** head response (`response.clips` empty, so `next_cursor == nil` and
  `lower == 0`) is still authoritative: it prunes every existing id and clears every
  `pendingDeleteIDs` entry except finalize-in-flight ones. (This closes the "empty head
  prunes nothing" gap.) This authority is only sound because the Pi now fails an empty list
  **closed**: A4 makes `/v1/clips` return `503`, not an empty `200`, on a scan or list-task
  error, so an empty head can never mean "the Pi could not read the card." A `503` leaves the
  in-flight fetch unresolved (no `.clipsResponse(.success)`), so nothing is pruned.

`reconciledHead` then unions in the response clips and applies the suppression filter (it
wraps `merged`), and returns the **authoritative-absent set** it computed (present ids it
pruned, plus pending-delete ids the window confirmed gone) so the reducer can fold that
whole set into `suppressedClipIDs` and out of `pendingDeleteIDs` before storing the list.
After a head response with epoch `E` is applied, the reducer drops
`clipFinalizeEpoch` entries with value `< E` (a surviving clip is now a normal or paged
list entry; a deleted one is gone), so the map stays bounded to clips finalized during the
currently in-flight head request.

Reducer wiring for the carve-out: `.load`/`.refresh` do `state.headEpoch += 1` (the same
generation counter `pageResponse`/`loadMore` already gate on, now bumped at request-time rather
than on `.clipsResponse` success) and thread that epoch into `fetchEffect`, which echoes it in
`.clipsResponse(epoch:, ...)`.
`.clipFinalized(clip)` stamps `state.clipFinalizeEpoch[clip.id] = state.headEpoch` before
merging. Because `fetchEffect` cancels any in-flight head fetch (`cancelInFlight: true`), a
superseded response is dropped rather than delivered; the epoch carve-out additionally
keeps a *late* head response from ever pruning a clip that finalized after it was issued.

> Why unbounded-above with an epoch carve-out (not the prior "keep every id above the
> newest returned"): after reconnect the app may hold clip 10 whose `clip_removed` it
> missed, while the fresh head returns `[8, 7]`. The prior heuristic kept 10 forever
> (`10 > 8`); the epoch rule prunes it (`finalizeEpoch[10]` predates this request) while
> still preserving a clip 11 that a `.clipFinalized` inserted *during* the request.

`.pageResponse(epoch:, .success)` keeps plain `merged`: window-authoritative pruning of an
*already-paged* (below-head) region needs the request cursor threaded into the action and
is deferred (a full refresh re-loads the head and re-pages -- see Out of scope).

### B8. App tests

**Existing-test migration (do this first -- the B4 signature and State changes break the current
suite at compile, and two existing tests now contradict the design):**
- **`.clipsResponse` signature.** Every `.clipsResponse(...)` call site moves to
  `.clipsResponse(epoch:, ...)` -- across `ClipsFeatureTests`, `AppFeatureTests`,
  `HomeViewControllerTests`, and any `CameraSamples`/store helper that builds the action (mirror
  how `pageResponse(epoch:, ...)` is already threaded).
- **`loadEpoch` -> `headEpoch`, retimed -- two classes.** The field is renamed and its bump moves
  from `.clipsResponse` success (response-time) into the `.load`/`.refresh` send-closure
  (request-time); a direct `.clipsResponse` no longer mutates it.
  - *Mechanical* (rename + shifted literals): `loadFetchesClipsOnce` (asserts it on the response)
    and `loadMoreFetchesNextPageAndAdvancesCursor` (seeds `loadEpoch: 1`).
  - *Restructured, not relabeled* -- `stalePageResponseDoesNotOverwriteResetFrontier` and
    `successfulHeadLoadResetsPaginationFrontierAfterReconnectGap`. Both drive the frontier reset by
    sending `.clipsResponse` **directly**, relying on response-time to bump the epoch and thereby
    strand an in-flight `.pageResponse` tagged with the pre-bump epoch. Under the retiming that direct
    send no longer bumps, so the seeded epoch never advances: `stalePageResponseDoesNotOverwriteResetFrontier`'s
    stale `.pageResponse(epoch: 5)` would now guard `5 == 5` and **merge** (gutting the exact
    stale-page-suppression invariant the test exists to protect), and `successfulHeadLoadResetsPaginationFrontierAfterReconnectGap`
    would pass on shifted literals while silently ceasing to exercise "a head advances the epoch."
    Restructure each so the frontier reset advances the epoch through a real `.refresh`/`.load` send
    whose queued fetch resolves to the head response (the way production resets the frontier), leaving
    the in-flight `.pageResponse` straddling a genuine epoch advance so it is still dropped. Merely
    shifting the epoch literals is not a valid migration here.
- **New hot-path field assertions.** `.clipFinalized` now stamps `clipFinalizeEpoch`, head loads bump
  `headEpoch`, and a head response drops stale `clipFinalizeEpoch` entries -- so exhaustive-`TestStore`
  tests that drive those actions (`snapshotSeedsOnlineRecordingProjectionAndLoadsClips`,
  `loadFetchesClipsOnce`, `clipFinalizedPrependsAndDedupsRegardlessOfStatus`) add the new-field
  mutations to their closures.

  finalize-race coverage):

## Part C -- Records to update in the same change

- **New raspi ADR 17** -- `raspi/docs/design/17-2026-07-02-clip-delete.md` (next raspi
  seq is 17 -- 16 is `16-2026-07-02-storage-coordinator-segment-id-witness.md`;
  date >= 2026-07-02). One decision: add `DELETE /v1/clips/{id}` + the `clip_removed`
  event. Frame it as **extending ADR 16's coordinator**, not introducing any interim
  mechanism:
  - It re-reconciles ADR 02's "every endpoint is preview/read/pull; nothing on the
    record path" (state that user delete is off the record path, only touches
    finished/unlocked/below-floor segments, and fails with zero effect on the live
    recording before any unlink).
  - Clip delete is the write-ahead-delete mutation ADR 16 reserved a seam for:
    `StorageCoordinator::delete_finished_segment` runs under the coordinator's mutation
    mutex and durably raises `state/state.json` `high_water_seq >= id` **before**
    unlinking, satisfying ADR 16's rule that any segment-removing mutation persist the
    witness first (rollover ids exceed the persisted witness, so the write-ahead is
    what keeps a deleted id from being reissued). Record the durable-delete ordering
    (witness raise -> unlink(s) -> recording-dir `fsync`, acknowledged only after both
    are durable) as honoring ADR 03's "every directory mutation is followed by an
    `fsync()` of the containing directory."
  - Record the start/delete floor ordering accurately as an invariant, not a bug fix: in
    both backends the segment file for an id is created only *after* `Input::StartCommand`
    sets the recorder floor, so the in-mutex floor check plus that file-after-floor ordering
    already prevent a delete from unlinking an active segment. The reservation turn --
    `reserve_start_segment` commits the floor inside the *same* coordinator turn as the
    witness persist, and is the only way to obtain a start id -- makes that ordering
    *structural* (defense-in-depth no future start path can regress), not the closing of a
    currently-reachable hole. **Explicitly note there is no interim `segment_id_lock` and no
    second witness file**; the coordinator mutex is the single seam.
  - Record that `/v1/clips`, `GET /v1/clips/{id}` (`serve_clip`), and the coordinator's
    allocation scan now **fail closed** on a scan error -- `503` (start `500` for allocation),
    never an empty `200` or a fail-open `404` -- so client-side head reconciliation can treat an
    absent id as an authoritative deletion and `serve_clip` stops masquerading a transient scan
    fault as a `404`. A truthful empty (a missing rec dir) stays an empty `200`. This tightens
    ADR 02's list contract and is the Pi-side half of the app's delete-reconciliation guarantee.
  - Record the clip-mutation `503` semantics: unlike `serve_clip`'s read `503` (ADR 02: "`503`
    = a *transient* SD-card read/open/stat/seek error, the only retriable status"), a
    `DELETE /v1/clips/{id}` `503` may be **non-transient** -- a corrupt `state/state.json`
    witness or an unreadable rec dir both surface as `503` and need operator action, not a blind
    retry. The *same* corrupt witness surfaces as a `500` on the start/allocation path
    (`BackendError::Storage`); delete deliberately keeps `503` rather than special-casing the
    witness to `500`, because the delete path already returns `503` for another persistent
    condition (an unreadable rec dir, A5), so one storage-fault status is more internally
    consistent than splitting corruption out. The app surfaces either to the user (re-insert +
    `.failed`, B4) rather than auto-retrying, so the distinction creates no retry loop.
  - Forward obligations: refuse incident-protected (`locked`) segments once locking
    exists; `If-Match` as a considered-but-deferred alternative. (Coordinator routing
    is already done, not a forward obligation.)
  - Add the endpoint + event to ADR 02's surface (or note the supersede). Add the ADR
    17 bullet to the ADR index in `raspi/AGENTS.md` ("Design decisions (ADRs)" list,
    after the ADR 16 entry). Run `just adr-check`.
- **Roadmap** -- `docs/roadmap.md`: add a swoop for user clip deletion (propose
  codename `ebb` -- confirmed unused in the main list and `#Icebox`). Note its
  dependency direction on `lime`/`kelp` and that it builds on the ADR 16 coordinator.
- **Contract README** -- `contract/events/README.md` event-type list gains
  `clip_removed`.
