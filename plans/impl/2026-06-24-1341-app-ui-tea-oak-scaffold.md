# Plan: app UI-architecture ADR + bespoke-TEA `oak` scaffold

## Context

Dan is ready to start the iPhone app (boilerplate + initial architecture). Two
decisions need recording before code lands (project convention: ADRs precede code):

1. **Drop SwiftUI for UIKit (programmatic).**
2. **Use a bespoke, zero-third-party-dependency Elm Architecture (TEA)** rather than
   a library (TCA / Mobius / ReSwift).

The decisive and unusual force is that **all development on this project is done by
LLMs**, which changes the calculus: a small, fully-in-context, self-consistent
architecture beats a large external API surface that churns across versions. The
stated optimization targets are **testability and traceability**.

`app/AGENTS.md` currently names "Swift, SwiftUI" as provisional direction; the Xcode
project is now generated (UIKit/Storyboard, Swift Testing) but holds only template code.
The next roadmap task is swoop `oak`'s app half: "App can call the mock Pi health
endpoint." This plan records the architecture decision as an ADR, makes the surrounding
docs consistent, then scaffolds the bespoke-TEA core plus the `oak` health-check slice
against the already-running mock service.

**Scope chosen by Dan:** ADR + doc updates **and** scaffold the code. Xcode project
already generated by Dan (see "Generated project" below). Persistence is out of scope.
Per Dan's instruction, **do not introduce a documented "dependency policy"** -- the
zero-dep property is stated as a plain consequence of the bespoke choice, not elevated
to a principle.

## Part 1: ADR `03` (the WHY-heavy deliverable)

**New file:** `app/docs/design/03-2026-06-24-app-ui-architecture.md`
(next app number is `03`; house style confirmed: `# ADR:` title, then
`Status/Date/Owner/Related` bullets, then `Context -> Decision -> Consequences ->
Alternatives considered`; plain ASCII; direct tone; alternatives as one-to-two-sentence
why-rejected bullets; ~120-180 lines is acceptable here given the WHY is the point).

- **Status:** Accepted | **Date:** 2026-06-24 | **Owner:** app
- **Related:** root `AGENTS.md`; `app/AGENTS.md` (Tech section it supersedes);
  `app/docs/design/02-2026-06-22-app-pi-transport-and-api.md` (the hand-rolled client
  becomes effect handlers behind the dependency boundary).

**Context (forces -- the part that matters most):**
- All development is LLM-driven (decisive). LLMs are strongest with small, in-context,
  self-consistent code and weakest against large, version-churning external APIs.
- Testability + traceability are first-class goals.
- App scope is narrow and well understood (a few screens, a socket client, a custom
  video/preview surface, CarPlay templates).
- The hardest surfaces are UIKit-shaped already: custom MJPEG frame rendering, AVPlayer
  + loopback-HLS playback (ADR 02), and CarPlay's `CPTemplate` API (UIScene/UIKit world,
  not SwiftUI).
- The generated project already enables Xcode 26 "approachable concurrency"
  (`SWIFT_APPROACHABLE_CONCURRENCY = YES`, all targets; SE-0461
  `NonisolatedNonsendingByDefault`) and main-actor-by-default
  (`SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor`, app target). The architecture is designed
  to these defaults, which makes a main-actor-resident store the path of least friction
  and removes most of the `Sendable` ceremony a pre-6.2 design would carry.

**Decision (state plainly):**
- **UIKit, programmatic, no SwiftUI.**
- **Bespoke minimal TEA:** pure
  `reduce(inout State, Action, Dependencies) -> Effect`, a `@MainActor`
  `Store<State, Action, Dependencies>` generic over a **struct-of-closures** dependency
  bag (the core knows no concrete dependency type), effects as data, and a hand-written
  generic `TestStore`. ~200-300 lines, owned in-repo.
- **Zero third-party dependencies** for the architecture (stated as a property, not a
  policy).
- **Concurrency posture: main-actor-resident.** The `Store` is `@MainActor` and effect
  operations run on the main actor (an ordinary `async` effect closure is
  `nonisolated(nonsending)` under approachable concurrency, so it runs on the caller's --
  i.e. the store's -- actor). There is no cross-actor hop on the action path, so almost no
  `Sendable` ceremony is needed and ordering stays deterministic. `@concurrent` (and the
  `Sendable` it then requires) is reserved for the genuinely off-main work that arrives
  later -- MJPEG frame decode and ranged clip pull (ADR 02 / swoop `fox`).

**Alternatives considered (each: crisp why-rejected):**
- **SwiftUI (UI layer).** Rejected: its implicit observation/state model duplicates and
  fights TEA's single-source-of-truth store; current SwiftUI idioms churn release to
  release (Observation/`@Observable`/NavigationStack), which hurts LLM codegen the same
  way TCA does; the app is small so SwiftUI's rapid-prototyping payoff is low; the
  hardest surfaces (preview, playback, CarPlay) are UIKit-shaped anyway.
- **The Composable Architecture (TCA).** Rejected: its public API has churned across
  several incompatible eras (`Reducer`/`pullback` -> `ReducerProtocol` -> `@Reducer`
  macro + `@ObservableState`; `ViewStore` optionality), so LLM-generated code routinely
  splices eras and fails to compile -- actively hostile to an all-LLM workflow. Also a
  large, SwiftUI-leaning dependency we would mostly not use. (Its `Effect` + dependency
  ideas are borrowed as reference.)
- **Mobius.swift (Spotify).** Rejected: a third-party dependency for ~200 lines we can
  inline and fully own; thinner LLM training footprint means more doc-dependence. We
  take its `Init`/`Update`/`Effect` design as a reference, not a dependency.
- **ReSwift.** Rejected: stale (last release Jan 2023); risky foundation for greenfield.
- **MVVM / MVC / VIPER / Clean Swift.** Rejected: no single source of truth and side
  effects are not modeled as data, so weaker testability and weaker traceability than a
  reduced action stream.
- **An elaborate home-grown framework.** Rejected: the goal is a *small, standard* TEA,
  not a novel architecture; LLMs do worse with idiosyncratic invented patterns.

**Consequences:**
- *Easy:* unit-testing logic (pure reducers + stubbed dependency closures +
  `TestStore`); traceability (one `send(_:)` choke point logs every action + state
  diff); swapping transport (URLSession now -> pinned `NWConnection` at swoop `fox`)
  behind the dependency closure without touching reducers.
- *Hard / risky:* we own the **effect-runtime correctness** (action serialization,
  reentrancy, `Task` cancellation + lifetime) -- the hard 10% libraries already solved.
  The main-actor-resident posture (above) *removes* most of the isolation/`Sendable`
  burden rather than adding it. **Mitigation:** keep the core tiny and standard, test it
  hard with a deterministic injected clock, and gate it through `swift-concurrency-pro`.
  Per-screen UIKit boilerplate is higher than SwiftUI; accepted given small scope.
- *Follow-ups (out of scope here):* persistence (SwiftData vs files) is a separate
  future ADR; the pinned-`NWConnection` client (ADR 02 / swoop `fox`) plugs into the
  `HealthClient`-style dependency boundary.

**Illustrative core sketch (mark as illustrative, not final):** compact pseudo-Swift
for `Effect<Action>` (including the cancellable `.run(id:cancelInFlight:operation:)`
case, whose `operation` is an ordinary `async` closure -- main-actor-resident -- that
feeds actions back through an `async` `send` re-entering the store on the **same** actor,
no cross-actor hop), the generic `@MainActor Store<State, Action, Dependencies>` (`send`,
observe, token-based effect tracking + id-keyed cancellation, `[weak self]` task bodies),
the pure `reduce` signature (no sender), the app's dependency bag of plain closures
defined *outside* the core, and a ~6-line `TestStore` usage example showing `send`/
`receive` assertions. State the posture explicitly: effects run on the main actor and
`@concurrent` is reserved for later off-main work.

**TEA core (`Architecture/`)** -- standard, minimal, dependency-agnostic, no deps.
**Swift 6 concurrency contract (main-actor-resident):** the `Store` is `@MainActor` and
the `.run` `operation` is an ordinary `async` closure -- under
`SWIFT_APPROACHABLE_CONCURRENCY` that makes it `nonisolated(nonsending)`, so when the
`@MainActor` store launches it the operation runs **on the main actor**, not off it.
Consequences: there is no cross-actor hop on the action path; the operation's captures and
`Action` itself do **not** need to be `Sendable`; the dependency closures it captures do
not need `@Sendable`. So the core carries **no `Action: Sendable` constraint** and **no
`@Sendable` annotations** -- a smaller surface, which is the point. `State` likewise stays
main-actor-isolated. The exact minimal annotation set is confirmed at implementation time
under approachable concurrency and gated through `swift-concurrency-pro`; `@concurrent`
(and the `Sendable` it would then force) is introduced only when real off-main work lands
(MJPEG decode / ranged pull at swoop `fox`).
- `Effect<Action>`: an enum --
  - `.none`
  - `.run(id: AnyHashable? = nil, cancelInFlight: Bool = false, operation: (_ send: (Action) async -> Void) async -> Void)`
  - `.cancel(id: AnyHashable)`

  Effects are data; the reducer captures whatever dependency closures the `operation`
  needs (deps are *not* stored on `Effect` or on the core). The `operation` is
  main-actor-resident (above), and the `send` callback handed to it is **async**
  (`(Action) async -> Void`): the operation does `await send(action)` to re-enter
  `reduce`. Because the operation already runs on the main actor, that `await` is a
  **same-actor** re-entry (an ordered continuation, not a thread/actor hop), so action
  delivery stays deterministic -- *simpler* than a cross-actor design, not harder. No
  `@Sendable` is needed on the operation or the `send` callback.
- `Store<State, Action, Dependencies>`: `@MainActor final class`,
  **generic over the dependency bag** so the core has no app-specific type. Holds
  `state`, the `reduce: (inout State, Action, Dependencies) -> Effect<Action>`, the
  injected `dependencies`, an observer list, and the in-flight task bookkeeping (below).
  `send(_:)` runs `reduce` synchronously (serialized on the main actor -> no state
  races), notifies observers, then executes the effect. Reducers are pure and have **no
  sender**: they return effects, never call `send` -- so the only re-entry into `reduce`
  is via an effect's async `send` callback, which re-enters on the same (main) actor.
  - **Task bookkeeping:** every launched `.run` task is tracked under an internal,
    monotonic token (so id-less tasks are tracked too); a separate `[AnyHashable: token]`
    index maps a cancellation `id` to its token. Tasks remove their own token (and any id
    entry) from both structures on completion.
  - **No retain cycle:** the launched `Task` body captures the store **weakly**
    (`[weak self]`); the `send`-back callback and the self-removal-on-completion both
    guard `weak self` and no-op if the store is gone. Otherwise store -> task -> store
    would be a strong cycle and `deinit` could never run for a long-lived effect. With
    weak capture the owner (the VC) controls lifetime; when it releases the store, the
    store deallocates and any still-running task's follow-up `send` no-ops (its `weak
    self` is nil) -- so a stale effect can never write to a resurrected store.
  - `.run` with an `id`: if `cancelInFlight`, cancel and drop the prior task for that id
    first; then launch, recording token + id->token. The operation feeds actions back by
    `await send(action)`. `.run` without an `id` launches and records only a token
    (fire-and-forget; tracked under a token but with no id, so it is not individually
    cancellable -- it runs to completion, and at teardown its follow-up `send` simply
    no-ops via `[weak self]`; see the teardown note below).
  - `.cancel(id:)` cancels the task whose token the id index points at, and clears both
    entries.
  - **Teardown (no active `deinit` cancellation for `oak`).** Under complete checking a
    `@MainActor` class's `deinit` is `nonisolated` and cannot read the main-actor-isolated
    task bookkeeping, so the store does **not** cancel tracked tasks from `deinit`. For
    `oak` this is fine: the `[weak self]` task bodies (above) already guarantee
    correctness -- no retain cycle, and a post-teardown follow-up `send` no-ops rather than
    mutating a dead store; an in-flight `URLSession` request just runs to completion and
    its result is discarded. In-life cancellation (`.cancel(id:)`, `cancelInFlight`) still
    works because it runs from `send` on the main actor. If active teardown cancellation is
    ever needed, use an `isolated deinit` (SE-0371) or a `nonisolated`, `Mutex`-backed
    cancellation box the `deinit` can drain -- a deepening pass, not `oak`.

  `observe { state in ... }` drives the view controller. One log line in `send` (action
  + state diff) = traceability.

**`oak` health slice:**
- `HealthResponse` (`Networking/`): `Codable, Equatable, Sendable` mirroring
  `raspi/service/src/health.rs` -> fields `boot_id`/`uptime_s`/`recording`/`t_ms`; decode
  with `JSONDecoder.keyDecodingStrategy = .convertFromSnakeCase` (-> `bootID? -> bootId`,
  `uptimeS`, `tMs`). Optionally read `X-Dancam-Proto`/`X-Dancam-Boot-Id` headers later.
- `HealthClient` (`Networking/`): a struct of plain closures --
  `var fetch: () async throws -> HealthResponse`, with
  `static func live(baseURL:transport:)` where `transport` is an injectable
  `(URLRequest) async throws -> (Data, URLResponse)` defaulting to
  `URLSession.shared.data(for:)`. `live` builds the `/v1/health` request, maps non-2xx
  status to a thrown `HealthError.http`, and decodes the body (decode failures ->
  `HealthError.decoding`). **Cancellation passes through untouched:** `live` rethrows
  `CancellationError` and `URLError(.cancelled)` **unchanged** and only wraps *other*
  transport errors into `HealthError.transport` -- otherwise a normal reload cancellation
  would arrive at the effect as `.transport(...)` and render `.failed`. The injected
  `transport` is the testable seam (no `URLProtocol` subclass needed). URLSession is fine for `oak`
  (localhost/LAN, no Wi-Fi interface pinning); the pinned `NWConnection` client arrives
  at swoop `fox` and swaps in behind the `fetch` closure. Default base URL
  `http://127.0.0.1:8080` (matches `just raspi-run`). (No `@Sendable`/`Sendable` ceremony:
  the effect operation that captures `fetch` is main-actor-resident, so the closures stay
  plain; add `@Sendable` only if a `@concurrent` effect ever captures them.)
- `HealthError` (`Networking/`): a small concrete `Error` + `Equatable` enum -- e.g.
  `.http(Int)`, `.transport(String)`, `.decoding(String)`. A typed error (not `any Error`)
  keeps the action's `Result` `Equatable` for `TestStore` assertions and makes the
  `.failed` display `String` an explicit mapping; `.failed`'s string is derived from it.
- `AppDependencies` (`App/`): an app dependency bag holding `health: HealthClient` (extend
  later). Lives in the app layer, not in `Architecture/`. No `Sendable` conformance is
  required -- the reducer captures its closures into a main-actor-resident effect operation.
- `HealthFeature` (`Features/Health/`): `State` = `.idle | .loading |
  .loaded(HealthResponse) | .failed(String)`; `Action` (`Equatable`, for `TestStore`
  `receive` assertions) = `.onAppear |
  .reload | .healthResponse(Result<HealthResponse, HealthError>)`. `reduce(&state,
  action, deps)` sets `.loading` and returns `.run(id: "health", cancelInFlight: true)`
  whose operation calls `deps.health.fetch()` and `await send(.healthResponse(...))`.
  **Cancellation:** if the operation was cancelled -- `Task.isCancelled`, a thrown
  `CancellationError`, or `URLError(.cancelled)` -- it returns **without sending any
  action**, so a fetch cancelled by `cancelInFlight` (the common reload-races-reload
  case) or by teardown never overwrites the newer `.loading` / `.loaded` with `.failed`.
  This depends on `live` having rethrown the cancellation unchanged (above) rather than
  wrapping it as `HealthError.transport`; the effect's catch matches on the raw
  cancellation signals, not on `HealthError`.
- `HealthViewController`: programmatic UIKit; creates the `Store`, `observe`s state into
  labels (status / decoded fields / error) and a "Reload" button that sends `.reload`.
- `SceneDelegate`: build `UIWindow`, set `HealthViewController` (in a nav controller) as
  root.
- **Info.plist:** add `NSAppTransportSecurity -> NSAllowsLocalNetworking = true` (allows
  cleartext to loopback/private IPs -- this is what the `oak` loopback run needs).
  `NSLocalNetworkUsageDescription` is **forward-looking**: the Local Network permission
  prompt fires only for unicast to a private LAN IP on a real device (swoop `fox` / the
  device path), **not** for loopback `127.0.0.1` and **not** in the simulator -- so it is
  not exercised by the `oak` simulator run. `NSBonjourServices` is deferred to discovery
  (swoop `fox`).

**Tests (`app/DanCam/DanCamTests/`, Swift Testing, `@testable import DanCam`):**
- `Support/TestStore.swift`: minimal `@MainActor` harness, generic over `Dependencies` --
  inject a stub dependency bag, run `reduce` synchronously, assert state after each
  `send`, and `await` + assert effect-fed actions (`receive`). Deterministic (stubbed
  closures; no real network or clock). (`@MainActor` because `DanCamTests` does not inherit
  the app target's main-actor default -- see Step 0.)
- `Architecture/StoreTests.swift`: exercise the **production `Store` runtime** directly
  (the hard part the `TestStore` bypasses), as async `@MainActor` Swift Testing cases,
  using continuations / `AsyncStream` to observe ordering rather than `sleep`:
  - `observe` fires with the new state after a state-changing `send`;
  - an effect-fed action is delivered back and applied (the `.run` -> async `send` path
    works);
  - a `.run` effect that immediately `await send(...)`s a follow-up action drives an
    ordered chain of state transitions deterministically (this is the supported
    reentrancy shape -- reducers never call `send`, so there is no reducer-reentrancy
    path to test);
  - `.cancel(id:)` and `cancelInFlight` stop in-flight work (assert the cancelled task's
    follow-up action never arrives);
  - an id-less `.run` task does not retain the store: with an effect in flight, hold the
    store through a `weak` reference, drop the last strong reference, and assert the store
    is actually deallocated (the `weak` ref becomes `nil`) -- proving no task->store retain
    cycle, and that a post-teardown follow-up `send` no-ops (`weak self`) rather than
    mutating a dead store. (`oak` does not assert active task cancellation at `deinit`;
    see the Store teardown note.)
- `Networking/HealthClientTests.swift`: drive `HealthClient.live` through a stubbed
  `transport` closure with canned `/v1/health` payloads -- assert snake_case decoding
  into `HealthResponse`, the request path/method, that a non-2xx status throws
  `HealthError.http`, and that a `transport` throwing `URLError(.cancelled)` is
  **rethrown unchanged** (it surfaces as `URLError(.cancelled)`, not wrapped into
  `HealthError.transport`).
- `HealthFeatureTests.swift` (`@MainActor`): reducer transitions via the `@MainActor`
  `TestStore` --
  `.onAppear`/`.reload -> .loading`; success -> `.loaded(expected)`; a `HealthError` ->
  `.failed(...)`; **and a cancelled fetch sends no action and leaves the state at
  `.loading`, never `.failed` -- covered for both signals: a stub `fetch` that throws
  `CancellationError` and one that throws `URLError(.cancelled)`** (the latter is the
  realistic URLSession case and is the one a naive `HealthError.transport` wrap would
  regress). (Behavioral and structure-insensitive: assert observable state, not
  internals.)
