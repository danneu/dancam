# Plan: Pin the dancam AP to WPA2-AES (kill the iOS "Weak Security" warning)

## Context

When an iPhone joins the Pi's dev access point (`dancam-dev`), iOS shows a
**"Weak Security -- WPA/WPA2 (TKIP) is not considered secure"** warning. The
cause is the AP profile: `README.md` Section 6 sets
`802-11-wireless-security.key-mgmt wpa-psk` but never pins the cipher, so
NetworkManager (via wpa_supplicant in AP mode) advertises the maximally-
compatible mix -- WPA1+WPA2 with **both TKIP and CCMP**. iOS sees TKIP in the
beacon and flags it.

The fix is to pin the AP to **WPA2-AES only** (RSN proto, CCMP pairwise+group).
This drops TKIP and WPA1 entirely and produces clean WPA2-AES, which iOS shows
with no warning (WPA2 Personal / AES is the mode Apple accepts when stronger
modes are not used; the verification below confirms it joins and serves on the
actual unit). It is the correct config regardless of the warning -- TKIP should
not be in the beacon.

This does not reverse ADR 06's WPA2-PSK decision; it specifies the cipher that
"WPA2" already implied. WPA3-SAE is deliberately **not** adopted here: it is out
of scope for this warning fix and would require a separate AP-mode validation
pass on the Zero 2 W (its behavior in NetworkManager/wpa_supplicant AP mode on
this chip is unproven in this repo -- no local test log backs it either way).
The security delta over WPA2-AES-with-a-strong-PSK is also negligible for a
local 2.4 GHz link carrying only preview + clip pull where the SD card is the
source of truth.

This change is **docs/runbook only** in the repo (the actual cipher properties
live in the on-Pi NetworkManager profile, applied via `nmcli`). Per the repo
convention, any change to real Pi provisioning must update `README.md` in the
same change, and the AP decision is owned by ADR 06.

## Out of scope / follow-up

- **Per-unit random SSID/PSK + QR onboarding (the real security win).** The
  cipher fix here removes a UX wart; the password is still a single shared,
  hand-typed dev secret. The substantive hardening -- a unique strong random PSK
  per unit, provisioned and delivered by QR -- is **deferred** and tracked as
  Icebox swoop `wren` (added by this change) and in ADR 06's Consequences. Do
  **not** attempt it here; it belongs to the production/car image, not the dev
  AP.
- **WPA3-SAE.** Out of scope for this fix (see Context); adopting it would need
  its own AP-mode validation pass on the Zero 2 W. Revisit only if a future move
  to a custom hostapd setup happens for other reasons -- ADR 06 already lists
  hostapd/dnsmasq as rejected-for-now.
- **Dev/prod AP split.** Production needs a persistent, autoconnecting AP
  (opposite of this dev profile's `autoconnect no` + restore-timer). That is its
  own future ADR + swoop, not part of this change.

## Verification

End-to-end (run after the repo edits are approved/applied):

1. **Apply to the already-provisioned dev Pi** (one-time remediation; the
   runbook now produces this on fresh setups). Over SSH on the Pi:
   ```sh
   sudo nmcli connection modify dancam-ap \
     802-11-wireless-security.proto rsn \
     802-11-wireless-security.pairwise ccmp \
     802-11-wireless-security.group ccmp
   ```
   `modify` only updates the stored profile -- it does **not** change a live
   AP's beacon until reactivation. The change takes effect on the next
   `nmcli connection up dancam-ap` (or `nmcli device reapply wlan0`); if the AP
   is already up, bring it down/up. Carry this caveat in the README runbook prose
   too, so an operator who checks their iPhone before reactivating does not read
   the still-present warning as a failed fix.
2. **Confirm the profile** shows the pinned cipher:
   ```sh
   nmcli -f 802-11-wireless-security.key-mgmt,802-11-wireless-security.proto,802-11-wireless-security.pairwise,802-11-wireless-security.group connection show dancam-ap
   # expect: wpa-psk / rsn / ccmp / ccmp
   ```
   Reconcile the README expected-security block to this actual output before
   finalizing the docs (see Scope section 1 -- the committed block is provisional
   until observed here).
3. **Bring the AP up** (use the systemd restore-timer pattern from README
   Section 6 so SSH-drop recovery is armed), then on a physical iPhone:
   - **Forget the old network first:** Settings -> Wi-Fi -> info on
     `dancam-dev` -> "Forget This Network", so iOS does not carry stale
     security info for the SSID and the warning check is unambiguous.
   - Rejoin `dancam-dev`.
   - **The "Weak Security" warning is gone** (the success criterion).
   - Re-confirm the link still works: fetch
     `http://10.42.0.1:8080/v1/health` (or `http://dancam.local:8080/v1/health`)
     and get `200` -- proves the cipher change did not break association/DHCP.
4. **Docs sanity:** run `just adr-check` (ADR filenames unchanged by an
   in-place amendment, so it stays green) and eyeball the README/roadmap diffs.

Success = iPhone joins `dancam-dev` with no weak-security warning and still
gets a `200` from the service.
