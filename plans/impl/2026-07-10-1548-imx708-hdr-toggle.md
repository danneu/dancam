# Plan: `just raspi-hdr on|off` -- dev toggle for IMX708 on-sensor HDR

## Context

The Arducam IMX708 supports on-sensor HDR (3 exposures combined on the sensor). It is
enabled via the v4l2 `wide_dynamic_range` control (id `0x009A0915`) on the sensor's
`/dev/v4l-subdev*` node, and the control only takes effect while the camera is closed --
libcamera re-enumerates sensor modes on next open. With HDR on, the sensor's only mode
is 2304x1296@30, which still covers the project's 1080p30 recording ceiling, so nothing
in the camera child, encoder, or preview changes.

Dan wants a development-time way to flip HDR on/off to compare picture output -- no app
toggle, no `/v1/settings` API yet (that remains the eventual durable design; ADR 02
already reserves an `hdr` settings field). The simplest honest tool is a `just` recipe
that, on the Pi, stops `dancam`, writes the control, restarts the service, and confirms
the camera actually came back.

Known behavior to document: the control resets to **off on reboot** (driver state, not
persisted), which is fine -- convenient, even -- for an experiment.

## Design (durable, tested; follows the `partition-card` pattern)

The repo already has a proven shape for "root-only Pi operation, hardware-free
regression, thin `just` wrapper": `raspi/scripts/partition-card.sh` +
`raspi/scripts/partition-card-test.sh` + `Justfile#raspi-partition` /
`#raspi-partition-test`. This plan mirrors it exactly, which keeps the fragile shell
orchestration out of the Justfile (no nested SSH-string quoting) and makes the cleanup /
readiness behavior directly testable.

### 1. `raspi/scripts/hdr-set.sh` -- the on-Pi operation (new)

Runs **as root**, invoked via `sudo bash` over SSH (same convention as
`partition-card.sh`, so it calls `systemctl` / `v4l2-ctl` / `curl` directly -- no
internal `sudo`). Takes one arg, `on|off`; the script owns mode validation so the test
can exercise it.

Logic:

1. `set -euo pipefail`. Map arg: `on`->`1`, `off`->`0`, else print `usage: hdr-set.sh on|off`
   to stderr and exit non-zero.
2. Guard `command -v v4l2-ctl` (see change 4 -- it is now a provisioned dep; the guard is
   a clear failure message, not the primary install path).
3. Discover the sensor subdevice by name -- the HDR control lives on the sensor subdev,
   not `/dev/video*`. Scan `${DANCAM_V4L_SYSFS_ROOT:-/sys/class/video4linux}/v4l-subdev*/name`
   for a name matching `^imx708` (matches both `imx708` and `imx708_wide`), map the sysfs
   dir back to `/dev/<basename>`. Error out if none found ("is the camera detected?").
   The `DANCAM_V4L_SYSFS_ROOT` env seam exists so the hardware-free test can point at a
   fake sysfs tree; documented in a comment.
4. Install the traps **before** stopping the service (in spirit from
   `Justfile#raspi-reset-data`'s trap -- see "Trap shape" below for the exact split),
   then:
   - `systemctl stop dancam`
   - `v4l2-ctl -d "$SUBDEV" --set-ctrl wide_dynamic_range=$VALUE`
   - read back and print `--get-ctrl wide_dynamic_range` so the applied state is visible.
5. `cleanup` runs on every exit path -- success, `set -e` failure, **and an interrupt
   after the stop** (see "Trap shape" below):
   - first line captures the entry status (`local rc=$?`), then **disarms `EXIT` and
     ignores the signals for the recovery window** in one step:
     `trap - EXIT; trap '' INT TERM HUP`. The `trap '' ...` (ignore, not reset) is
     load-bearing: the signal handlers are **not** one-shot, so a second INT/TERM/HUP
     arriving during `systemctl start` or the readiness poll would otherwise fire the
     handler again, `exit` immediately, and abandon recovery with the EXIT trap already
     gone -- leaving `dancam` down. Ignoring the signals makes the bounded recovery
     uninterruptible; `rc` (the signal-derived status) is preserved for the final re-exit.
   - `systemctl start dancam` (exactly one restart attempt); if it fails, print
     "recording is DOWN" and exit non-zero.
   - **Readiness gate (F1 fix):** bounded poll of `curl -fsS http://localhost:${DANCAM_PORT:-8080}/v1/status`
     until it reports `camera_state: running`, not merely HTTP 200. Grep tolerantly for
     `"camera_state"[[:space:]]*:[[:space:]]*"running"`. On timeout
     (`${DANCAM_HEALTH_TIMEOUT:-60}`s) print "recording is DOWN" and exit non-zero.
   - otherwise re-exit preserving the status `cleanup` was entered with (the `rc`
     captured on its first line) -- see "Trap shape" for why that status is trustworthy.

   **Trap shape (F1) -- `cleanup` on EXIT only, signals set the status.** The subtlety
   the previous draft got wrong: a handler wired directly to `INT`/`TERM`/`HUP` captures
   `$?` from *whatever command last ran*, which after a completed `systemctl stop` /
   read-back is very likely `0`. If `cleanup` then restarts successfully and re-exits that
   captured `0`, an interrupted toggle reports **success** -- the exact false-success trap
   `raspi-reset-data` documents. Fix: register `cleanup` on `EXIT` only, and register a
   tiny separate handler on `INT TERM HUP` whose whole job is to `exit` with an explicit,
   signal-derived non-zero status (`130` INT / `143` TERM / `129` HUP, i.e. `128+signum`).
   That `exit` then invokes the single `EXIT` `cleanup`, which restarts once, gates
   readiness, and -- because it re-exits the status it was entered with -- **preserves the
   signal-derived failure code**. So the restart-on-interrupt guarantee is robust and
   directly testable, and an interrupt can only ever make the script report failure, never
   a false success. This is the deliberate, safe divergence from `raspi-reset-data`
   (which relies on bash running the EXIT trap on an untrapped fatal signal): we trap the
   signals explicitly *and* keep the status provenance `raspi-reset-data` was protecting.

Rationale for `/v1/status`+`camera_state: running` over `/v1/health` 200 (per review F1):
`health` returns 200 whenever the HTTP layer is up, and its `recording` field is `false`
after any restart because the service does **not** auto-record on boot (recording starts
only via `POST /v1/recording/start` -- see `raspi/service/src/recording.rs`). So
`recording: true` would be a false *failure*, and health-200 alone would be a false
*success* while the camera child crash-loops. `camera_state` (`world.rs#CameraState`,
`#[serde(rename_all = "snake_case")]` -> `starting|running|restarting|offline`) reaching
`running` is the true "camera reopened with the new sensor mode" signal -- exactly what an
HDR toggle must confirm. Verified value string against `contract/events/snapshot.json`
(`"camera_state": "running"`).

### 2. `raspi/scripts/hdr-set-test.sh` -- hardware-free behavioral test (new)

Mirrors `partition-card-test.sh` (runs on macOS, no Pi/root). Prepends a temp dir of mock
executables to `PATH` and points `DANCAM_V4L_SYSFS_ROOT` at a fake tree. Mocks:
`systemctl` (records stop/start calls, configurable to fail start), `v4l2-ctl`
(configurable to fail `--set-ctrl`; `--get-ctrl` echoes a value), `curl` (configurable to
return a `camera_state` that is `running` after N polls, or never). Fake sysfs:
`.../v4l-subdev0/name` containing `imx708_wide ...`.

Cases (the substantive behaviors the review flagged as untested):

- `on` -> writes `wide_dynamic_range=1`, stops then starts dancam, polls to `running`, exit 0.
- `off` -> writes `=0`, same lifecycle, exit 0.
- invalid arg (`bogus`, missing) -> usage error, exit non-zero, **no** `systemctl stop`.
- subdevice missing (empty fake sysfs) -> clear error, exit non-zero, **no** stop.
- `v4l2-ctl --set-ctrl` fails -> dancam is still restarted by the trap (exactly one start),
  exit non-zero.
- **`systemctl start` fails (F2)** -> "recording is DOWN" on stderr, exit non-zero, and
  exactly one restart attempt (mock counts `start` invocations).
- **interrupt after stop (F1/F2)** -> the mocked `systemctl stop` (or the mocked
  `v4l2-ctl --set-ctrl`, which runs next) sends SIGTERM to the script's PID; assert the
  trap still restarts dancam (exactly one `start`), gates on readiness, **and exits with
  the signal-derived status `143`** (not merely "non-zero", and specifically not `0`).
  Asserting the exact `128+signum` code is what pins down the F1 fix -- it proves the
  interrupted toggle reports failure and can never re-exit a stale `0` even though the
  restart+readiness path itself succeeded. This is the case that proves the primary safety
  contract under interruption. (A companion SIGINT variant asserting `130` is cheap and
  documents the `128+signum` mapping.)
- **re-interrupt during recovery (F1)** -> after the first SIGTERM drives `cleanup`, the
  mocked `systemctl start` (or the `curl` readiness poll) sends a *second* SIGTERM to the
  script's PID; assert recovery still completes -- exactly one `start`, readiness gated --
  and the script exits with the original signal-derived status (`143`), proving `cleanup`
  ran to completion exactly once. This is the case that proves `trap '' INT TERM HUP`
  makes the bounded recovery uninterruptible; without it the second signal would abandon
  recovery and leave `dancam` down.
- readiness never reaches `running` -> "recording is DOWN", exit non-zero (bounded, so
  the test sets a tiny `DANCAM_HEALTH_TIMEOUT`).

The `systemctl` mock records an ordered log of `stop`/`start` calls to a temp file so
each case can assert both the count and the order (stop-before-start, exactly one start).

### 3. `Justfile` -- thin wrapper + test recipe

- `raspi-hdr mode`: shebang bash recipe using the standard `HOST`/`SSH_KEY` (with `~`
  expansion) block; `scp` `raspi/scripts/hdr-set.sh` to `/tmp/dancam-hdr-set.sh`, then
  `ssh -t` runs `sudo bash /tmp/dancam-hdr-set.sh <mode>`. `DANCAM_PORT` /
  `DANCAM_HEALTH_TIMEOUT` are forwarded to the remote environment (see quoting below).
  No validation or logic in the recipe; the script owns mode validation (so behavior is
  covered by the test, not duplicated untested in Just). The `raspi-partition` shape.
  Doc comment: what it does, camera-closed requirement, 2304x1296@30 cap (still covers
  1080p30), resets-off-on-reboot caveat, `DANCAM_HOST` override.

  **Shell-safe boundary (F1).** `{{mode}}` is substituted by Just as raw text *before*
  the recipe shell parses, and `ssh` reconstructs its args into a string the remote
  login shell parses *again* -- two injection points a bad `mode` (e.g. `on;reboot`)
  could break out of. Close both, so the value always arrives at `hdr-set.sh` as one
  intact argument where the existing on|off guard rejects it (quoting is what makes that
  guard load-bearing rather than bypassable):
  - Bind the Just interpolation through Just's quoting at the top of the recipe:
    `mode={{quote(mode)}}` -- this neutralizes metacharacters at the Just->recipe-shell
    boundary.
  - Build the remote command so the value survives the second (remote) parse: pass it as
    a single-quoted token, e.g. `ssh ... "sudo bash /tmp/dancam-hdr-set.sh $(printf %q "$mode")"`,
    and likewise `printf %q`-quote any forwarded `DANCAM_PORT`/`DANCAM_HEALTH_TIMEOUT`
    and the `HOST`/`SSH_KEY`/remote-path values used to assemble `scp`/`ssh`.
  Defense in depth: even a value that somehow reached the script cannot do harm, because
  it is only ever compared against `on`/`off`; quoting ensures it also cannot break out
  before reaching that comparison.
- `raspi-hdr-test`: `bash raspi/scripts/hdr-set-test.sh` (mirrors `raspi-partition-test`).

`raspi-hdr` is now a documented operation, so `v4l2-ctl` must exist on a freshly

Add `v4l-utils` to the "Install the camera process runtime dependencies" task's `name`

### 5. `raspi/README.md` -- ops blurb

Per the repo rule (human-facing ops changes update the README in the same change): add a
short paragraph in section 8 ("Deploy and run the service"), after the `raspi-reset-data`
paragraph (anchor: the paragraph beginning "Leaving old-format segments in place is
harmless"). Content: `just raspi-hdr on|off` flips the IMX708's on-sensor HDR for A/B
comparison; it stops `dancam` (the sensor only accepts the control while the camera is
closed), writes `wide_dynamic_range` on the sensor subdevice, restarts, and waits until
`/v1/status` reports `camera_state: running`; HDR caps the sensor at 2304x1296@30
(recording stays 1080p30); resets to off on reboot.

No ADR needed -- this is a dev tool, not a design decision; the eventual
`/v1/settings`-based toggle stays parked in ADR 02 / roadmap.

## Verification

Hardware-free (must pass at implementation time, on the Mac):

1. `just raspi-hdr-test` -- all cases above green, including the F2 start-failure and
   interrupt-after-stop restart guarantees.
2. `just raspi-provision-lint` -- site.yml still lints after the dep addition.
3. `just --list` parses the Justfile (both new recipes appear).
4. Injection boundary (F1): `just raspi-hdr 'on;touch /tmp/pwned'` against a throwaway
   or dry-run target must reach the script's usage guard and refuse -- the marker file is
   never created locally or on the Pi. (If no target is reachable, a wrapper dry-run that
   echoes the assembled `ssh` command instead of running it demonstrates the value stays a
   single quoted token.)

Against the real Pi on home Wi-Fi (`dancam.local` reachable), when hardware is up:

5. `just raspi-hdr on` -- read-back `wide_dynamic_range: 1`, restart, and the recipe only
   returns success once `camera_state` is `running`. Confirm the HDR mode was selected:
   `ssh dancam.local 'journalctl -u dancam -n 50 --no-pager'` shows the camera child
   starting cleanly (and a 2304x1296 sensor mode, if logged).
6. Eyeball output: `http://dancam.local:8080/v1/preview/live.mjpeg` or pull a fresh clip.
7. `just raspi-hdr off` -- read-back `wide_dynamic_range: 0`, camera back to `running`.
8. `just raspi-hdr bogus` -- usage error, exit non-zero, no service disruption.

Steps 5-8 depend on live hardware; if the Pi is down at implementation time, steps 1-4
gate the change and Dan runs 5-8 when the hardware is up.
