# Log SSE events as the raspi service emits them

## Context

Today the raspi service never logs the SSE events it produces. The `/v1/events`
path only logs on *problems* -- `tracing::warn!` when a subscriber's broadcast
buffer lags (`events.rs#fn events`) and `tracing::error!` when segment-duration
enrichment fails (`events.rs#fn enrich_current_segment`). When debugging what the
app receives (or does not receive), there is no server-side record of the event
stream, so you cannot tell whether the server produced an event or the link
dropped it.

Goal: emit one log line per SSE frame, so the log reflects what the server
*produces*, independent of how many subscribers are connected. Two frame kinds
need two log sites (see below): the sequenced deltas + heartbeat that flow
through the hub, and the per-connection `Snapshot` frame synthesized in the
handler. This is a diagnostic addition consistent with the existing tracing
design (ADR `raspi/docs/design/13-2026-07-01-request-logging-and-log-access.md`);
it reverses no prior decision, so no new ADR is warranted.

Decision already taken with the user: **non-heartbeat events at `DEBUG`,
`Heartbeat` at `TRACE`** (heartbeats fire every tick and are pure liveness, so
they must not flood a debug session).

## Where SSE frames originate (two log sites, not one)

Almost every frame is a **sequenced delta or heartbeat**: `event_hub.rs#fn drive`
assigns each `World::apply` output a `seq`, broadcasts it, and collects it. All of
these -- from HTTP command handlers, the background recorder loop, telemetry, and
the heartbeat task -- flow through this one loop (`backend.rs` drives them all via
`hub.drive_now(...)`), so logging there is complete and subscriber-count
independent for that class. `Event` is already imported at the top of
`event_hub.rs`.

The one exception is the **initial `Snapshot` frame**. It is not a
`World::apply` output and never passes through `drive`: `events.rs#fn events`
builds it per connection from `connection.snapshot`
(`Event::Snapshot(snapshot)`) as the first streamed frame. So it needs its own
log line in the handler -- one DEBUG line per `/v1/events` connect. This is the
single deliberate per-connection log outside the hub.

## Change A -- hub deltas + heartbeat (`event_hub.rs#fn drive`)

Inside the `for event in events` loop, after constructing `seq_event` and
before/around the existing `self.events_tx.send(...)`, add level-split logging:

```rust
// Heartbeat is pure liveness and fires every tick, so it stays at TRACE;
// real state changes log at DEBUG. Logged here, where every sequenced delta
// is broadcast, so it reflects what the server produces regardless of how
// many SSE subscribers are connected. (The initial per-connection Snapshot
// frame is logged separately in events.rs#fn events.)
if matches!(seq_event.event, Event::Heartbeat { .. }) {
    tracing::trace!(seq = seq_event.seq, event = ?seq_event.event, "emit");
} else {
    tracing::debug!(seq = seq_event.seq, event = ?seq_event.event, "emit");
}
```

Rendered under the existing `tracing_subscriber::fmt::init()` full formatter
(message-then-fields, same shape as the current request logs):

```
... DEBUG dancam: emit seq=207 event=SegmentOpened { session: 4, id: 32, at_ms: 361001 }
... DEBUG dancam: emit seq=210 event=ClipFinalized(ClipMeta { id: 32, start_ms: Some(361001), dur_ms: Some(59999), bytes: 71303168, locked: false, etag: "a1b2c3d4", time_approximate: false })
... TRACE dancam: emit seq=212 event=Heartbeat { t_ms: 425000 }
```

Events driven from inside an HTTP handler inherit that request's span
(`request{request_id=.. method=POST path=..}:`); events from the background
recorder/telemetry/heartbeat tasks render bare (`dancam: emit ...`).

## Change B -- per-connection snapshot frame (`events.rs#fn events`)

Log the first frame where it is built, reusing the already-constructed `Event`
value so no clone is needed (the `?`-Debug borrow happens before the value is
moved into `sse_frame`). `Event` is in scope (declared in this file). Same `emit`
message and `seq`/`event` fields as Change A, so the whole stream greps together:

```rust
let snapshot = enrich_current_segment(connection.snapshot, &state).await;
let first_event = Event::Snapshot(snapshot);
tracing::debug!(seq = connection.seq, event = ?first_event, "emit");
let first = tokio_stream::once(sse_frame(connection.seq, first_event));
```

Renders once per `/v1/events` connect. This `debug!` runs synchronously inside
the handler future (before the `Sse` response is returned), and `request_trace`
(`lib.rs#fn request_trace`) instruments `next.run(request).await` with the
request span -- so per ADR 13, handler-future logs are covered and this line
**inherits the GET request span** (only the later SSE response-body stream work
is uninstrumented):

```
... DEBUG request{request_id=84 method=GET path=/v1/events}: dancam: emit seq=42 event=Snapshot(Snapshot { recorder: RecorderSnapshot { phase: Idle, ... }, camera_state: Running, boot_id: "...", uptime_s: 362, storage: Some(...), temp_c: TempC { ... }, mem: Some(...) })
```

Snapshot is DEBUG (state, not liveness) and low-volume (one per connect), so it
does not warrant the heartbeat TRACE treatment.

## Enablement (no unit change)

The filter is the default `Targets` parser (no `env-filter` feature), which
supports `target=level`. The new lines therefore surface with the *already
documented* knob:

- `RUST_LOG=dancam=debug` -> per-event `emit` lines (plus existing request INFO logs)
- `RUST_LOG=dancam=trace` -> additionally heartbeat `emit` lines

Do **not** bake `RUST_LOG` into the shipped `raspi/dancam.service` -- default
INFO must stay quiet; a per-event line in the persistent journal on every state
change is exactly the flood the TRACE split avoids. Operators opt in per the
existing docs (`sudo systemctl edit dancam` drop-in, or a temporary run).

## Docs (small, operator-facing only)

Extend the two existing `RUST_LOG=dancam=debug` verbosity notes so operators know
what DEBUG/TRACE now include -- one clause each, no restructuring:

- `raspi/AGENTS.md` -- the verbosity bullet under `### Running` (the
  `RUST_LOG=dancam=debug` line): note DEBUG now logs each emitted SSE event
  (`seq` + body) and TRACE adds heartbeats.
- `raspi/README.md` -- the matching verbosity note at the tail of
  `## 7. Deploy and run the service`: same one-line addition.

No ADR change: ADR 13 already establishes `RUST_LOG` verbosity control; this adds
log statements within that design rather than changing it.

## Tests

No new test, for either change site. Per the repo's plan-review bar (behavioral,
structure-insensitive tests only), asserting log output would mean string-matching
formatter output against a captured subscriber -- structure-sensitive and low
value, and there is no log-capture test infra or `tracing-test` dependency to
build on. Both added calls are level-gated side effects: Change A does not alter
`drive`'s return value, and Change B does not alter the SSE frame stream (it reuses
`first_event` before moving it into `sse_frame`).

Guard instead by confirming existing behavior is untouched: the current
`event_hub.rs` tests that hammer `drive` heavily -- `lagged_receiver_reports_gap_...`
(258 drives) and `concurrent_connects_fold_...` (100 drives) -- must still pass.
Under `cargo test` there is no global subscriber, so the macros short-circuit
before the `?`-Debug format runs; no measurable cost.

   `/var/lib/dancam/rec` (not writable locally) and `MockRecorder::start` does

   cadence is `DANCAM_MOCK_SEGMENT_SECS` (default 5s) -- shorten it to see a
