# ADR: Clip delete endpoint and removal event

- **Status:** Accepted
- **Date:** 2026-07-02
- **Owner:** raspi
- **Related:** `02-2026-06-22-app-pi-transport-and-api.md` (wire contract);
  `03-2026-06-23-storage-ring-buffer-incident-lock.md` (future protected
  segments and ring GC); `16-2026-07-02-storage-coordinator-segment-id-witness.md`
  (storage coordinator and durable segment-id witness)

## Context

The app can browse and pull finished clips, but before this decision it could not
remove footage from the Pi. Adding delete is the first phone-initiated mutation of
already-committed footage, so it needs to be reconciled against ADR 02's original
"preview/read/pull only" framing and ADR 16's segment-id witness rule.

The operation is defensible only if it stays off the record path. Deleting a clip
must never touch an active segment, must never make a deleted sequence reusable, and
must not let a transient scan failure masquerade as an authoritative empty clip list.
Those properties are also required by the app's optimistic-delete reconciliation:
after a fresh head list omits an id, the app treats that absence as a confirmed
removal.

## Decision

Add `DELETE /v1/clips/{id}` for single finished-clip deletion and add the symmetric
`clip_removed` SSE event. A successful delete returns `204 No Content` only after the
filesystem mutation is durable, and only then emits `clip_removed { id }` through the
same ordered event stream as `clip_finalized`.

The delete is a storage-coordinator mutation, not handler-owned unlink logic.
`StorageCoordinator::delete_finished_segment` runs under the coordinator mutation
mutex and:

1. Checks the live recorder floor inside the mutex. Any id at or above the floor is
   treated as not deletable and returns `404`.
2. Scans the recording directory for every regular file whose parsed segment sequence
   matches the requested id. This intentionally collects bare and stamped duplicate
   names; reconstructing a single path or deduping would leave valid footage behind.
3. Reads `state/state.json` and durably raises `high_water_seq` to at least the id
   before the first unlink. A corrupt committed witness fails closed with no unlink.
4. Removes every matched path and fsyncs the recording directory before the handler
   acknowledges success.

This is the write-ahead-delete mutation ADR 16 reserved a seam for. There is no
interim `segment_id_lock`, no second witness file, and no alternate delete witness.
The coordinator mutex remains the single mutation seam, and `state/state.json`
remains the durable no-id-reuse witness.

Start/delete ordering is expressed structurally. In both backends, a segment file for
a new start id is created only after `Input::StartCommand` sets the recorder floor,
so the in-mutex floor check plus that file-after-floor ordering already prevents
deleting an active segment. `StorageCoordinator::reserve_start_segment` now commits
that floor inside the same coordinator turn as the witness persist, and it is the only
production way to obtain a start id. This is defense-in-depth: it prevents a future
start path from reordering file creation ahead of the floor, rather than closing a
currently reachable unlink race.

Delete failure semantics are layered:

- A floor block or "no matching regular file" returns `404`.
- A scan, witness, unlink, or directory-fsync failure returns `503`.
- A failure before unlink changes nothing on disk.
- A failure after unlinking begins may have removed one or more duplicate paths, but
  it does not emit `clip_removed` and does not return `204`.
- The safety invariant still holds after any post-witness failure: `high_water_seq >=
  id` was already made durable before the first unlink, so the deleted id cannot be
  reissued.

`DELETE /v1/clips/{id}` `503` is broader than the read path's retriable `503`. A
corrupt `state/state.json` witness or an unreadable recording directory can both
surface as delete `503` and may need operator action. The same corrupt witness still
surfaces as `500` on the start/allocation path through `BackendError::Storage`;
delete keeps one storage-fault status because it already uses `503` for persistent
directory faults. The app surfaces non-404 delete failures by re-inserting the row and
marking the clips state failed; it does not blindly auto-retry.

The clip list and single-clip lookup now fail closed on recording-directory scan
errors. `/v1/clips` returns `503` rather than an empty `200`, `GET /v1/clips/{id}`
returns `503` rather than a fail-open `404`, and start allocation propagates the scan
error instead of lowering the allocation floor. A missing recording directory remains
a truthful empty list. This tightens ADR 02's contract so a client can treat absence
from a successful head list as authoritative, while IO failure remains distinguishable
from deletion.

Mutation hardening is shared. The existing `Content-Type: application/json` and
non-empty `Idempotency-Key` checks move into a neutral mutation-header module used by
recording start/stop, time sync, and clip delete.

## Consequences

- The app gains Home swipe-to-delete and clip-viewer delete while every connected
  client can reconcile via `clip_removed`.
- The delete is off the record path: active or newly reserved segments are refused at
  the coordinator boundary before any scan or unlink.
- A deleted segment id is never reused, including the top finished rollover id whose
  sequence may exceed the previously persisted witness.
- Partial duplicate deletion can converge through a repeated DELETE or a fresh head
  list, but a `503` is intentionally not acknowledged as durable success.
- The app's head-list reconciliation is now sound because the Pi no longer returns a
  successful empty list when it failed to scan existing footage.
- Future incident locks must make protected clips undeletable. Until that layer
  exists, `locked` is always false and delete has no protection check to consult.

## Alternatives considered

- **Reconstruct only `seg_{id}.ts`.** Rejected. Stamped-only clips are valid, and bare
  plus stamped duplicates can coexist during the current flat-layout transition.
- **Use a separate delete witness.** Rejected. ADR 16 already made
  `state/state.json` the segment-id high-water floor; a second witness would split the
  no-reuse invariant.
- **Treat DELETE as idempotent success when already gone.** Rejected. Returning `404`
  preserves the existing clip-pull meaning of "not pullable or gone." The app still
  treats delete `404` as "keep the row removed."
- **Default no-op `Backend::note_clip_removed`.** Rejected. A backend that returns
  `204` but forgets to emit the reconcile event would be a silent correctness bug, so
  the backend hook is required.
- **Build `If-Match` now.** Deferred. Segment ids are never reused, so id alone is a
  stable delete target. An `If-Match` guard can be added later if the app needs a
  user-visible stale-row warning.
